# v0.11.2 — Infrastructure: remote py-spy evidence and teardown

The pod remains a disposable executor and `/workspace` is the persistent
runtime volume. Once the unchanged smoke has shown `ray::WorkerDict` without
progress for three minutes, install `py-spy` only into
`/workspace/verifierforge/.venv`; it is a diagnostic tool, not a project
dependency or requirements-file change.

The process manifest uses the recorded job PGID plus the verl/Ray session ID,
so it includes the detached launcher and all associated Python workers without
sampling unrelated pod processes. Each `py-spy dump` is saved verbatim under
`runs/<job>/evidence/` and synced to the ignored laptop `runs/` directory;
the same raw text is embedded in the run sheet for durable review.

At the timebox or after a failed retest, `vf kill <job>` is the mandatory
cleanup path. It must remove tmux, signal the recorded process group, run the
Ray fallback, and fail if GPU memory remains. A final `nvidia-smi` result of
`0 MiB` is the W1 acceptance check for this diagnostic job.
