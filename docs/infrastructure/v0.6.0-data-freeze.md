# v0.6.0 — data freeze and remote Gate B operations

**Status:** in progress. Linked plan: `docs/versions/v0.6.x/v0.6.0-data-freeze.md`.

## Data flow

Live augmentation and Gate A run on the laptop. Secrets remain in its ignored
`.env`; neither environment file nor API key is rsynced, committed, echoed, or
sent to the RunPod shell. Freeze evidence and the candidate dataset are Git
inputs only when they contain no secrets. Training metrics/logs travel through
Storage and `vf watch`; checkpoints stay under `/workspace/runs` and never enter
Git or normal watch synchronization.

## Gate B

After and only after a passing tagged freeze, invoke the existing detached
control plane: `bash scripts/vf train <job> <cfg>`. The RunPod remains a
stateless executor; it pulls the tag through its read-only deploy key and runs
inside tmux. `vf logs`, `vf status`, `vf kill`, and `vf watch` are the only
operational controls required for this one-hour smoke.

## Failure / rollback

A failing Gate A starts no pod job and reports its three measured numbers to the
human unchanged. A failed/aborted Gate B keeps its Storage metrics for diagnosis
but does not move or retag the frozen data. No remote changes are pushed from
the pod.
