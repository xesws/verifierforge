# v0.4.0 — Infrastructure: Disposable GPU and Local Sandbox

## RunPod Execution

The laptop remains the control plane and retains the OpenRouter key. The pod is
a disposable SSH/tmux executor; its durable inputs, staging checkpoints, and
logs live under `/workspace`. `vf watch` excludes both published `ckpt/` and
native `.verl-staging/` checkpoint trees (as well as safetensors), while a small
final manifest and curve image may sync back.

The proven pod virtual environment currently uses Python 3.12 even though
repository source targets Python 3.11. D2 deliberately uses that existing
`.venv/bin/python` with its verified verl/vLLM stack rather than rebuilding the
GPU environment during the smoke; source and unit tests remain 3.11-compatible.

## Sandbox Boundary

Verifier candidates execute only in a local Docker Python 3.11 image. Each run
has no network, read-only root filesystem/input mount, non-root uid, dropped
capabilities, `no-new-privileges`, 64 PIDs, one CPU, 512 MiB memory, a 64 MiB
temporary filesystem, a five-second host timeout, and 32 KiB captured output.
There is no host-Python fallback. Docker Desktop must be running before its
integration tests run.

## Validation

Test a safe candidate, blocked network/write attempts, and timeout handling.
Run the real launcher remotely only after the data and verifier fixtures freeze.

## Acceptance Evidence — 2026-07-15

The real run detached through tmux and survived the invoking SSH connection.
After the published step-50 checkpoint, `vf kill d2demo` removed the tmux
session and left no GPU process; the next `vf train` resumed from Storage. The
watcher copied public metrics and small artifacts only: local and remote JSONL
SHA-256 matched, while local `ckpt/` and `.verl-staging/` were absent.
