# v0.16.0 — durable run storage on S3

S3 settings use `VF_S3_BUCKET`, optional `VF_S3_PREFIX`, optional
`VF_S3_REGION`, and the explicit `VF_STORAGE_BACKEND=s3` selector. The
selector does not change the default local path. The pod-local cache has no
durability claim and can be removed after a node loss; all recoverable state
is reconstructed from committed S3 manifests and metric objects.

The real cloud check may perform one standard authenticated S3 round trip. A
single `AccessDenied`/403/scope error is preserved without secret fields and
is an `OWNER-ACTION`: attach `AmazonS3FullAccess` (or equivalent least-privilege
bucket permissions) to the credentials, then run the exact recorded command.
No policy is changed by this repository.

For the GPU proof, credentials are passed directly to the detached remote
process environment, never written to a pod file, and are cleared after the
process starts. Control-plane logs record only the fact of injection/clearance,
not names or values of secrets.
