# v0.12.7 — Pod-local export conversion and serving gate

All X1/X2/X3/X4 work happens on the H100 pod and its `/workspace` volume. The
converter must estimate or check free space before materializing bf16 merged
models, preserve original exports, and leave conversion evidence beside each
checkpoint. It uses only pod-local base-model cache and loopback vLLM; no model
download or external completion API is allowed.

X3 and future bridge smoke tests bind only to loopback, wait for `/v1/models`,
submit a short completion, and terminate their process group before continuing.
Failure preserves server logs and blocks publication/evaluation. X4 can use
ports 8011--8018 serially, never concurrently; its evidence and report remain
under the M3 run. M6 verifies remote/local SHA-256 parity after M5 passes.

X1 establishes that X2 materializes complete merged bf16 checkpoints rather
than tiny adapter files. Conversion must be atomic per checkpoint and leave an
incomplete destination absent or quarantined on failure, so M5 cannot consume a
partial directory.

The completed batch added 27 GiB to the M3 checkpoint tree (61 GiB to 88 GiB);
each output was atomically renamed only after its manifest and rewritten index
were present. No source export or model cache was deleted.

X3 used loopback-only vLLM on `127.0.0.1:8021`, returned HTTP 200 from
`/v1/models` and a real completion, and atomically retained the entire exchange
at `evidence/export-compat-v0127/x3-serving-smoke.json` (SHA-256
`e83e20ae11eeb7ce863851aeaf3afda207581183fc63c4cb7072fda029986039`). Its
temporary tmux session was then terminated; GPU use settled at the known 2,134
MiB host-namespace ghost allocation and 0% utilization before X4.

X4 completed sequentially and left GPU use at the same accepted 2,134 MiB
ghost allocation after its final server exited. X5's future publication smoke
will select a free loopback port, constrain vLLM's smoke allocation so it can
coexist with the rollout engine, kill its process group on either outcome, and
atomically retain its models/completion transcript before Storage publication.
M6 will sync only report/evidence/artifact metadata through `vf watch` and
verify their remote/local SHA-256 identities; it will not duplicate checkpoint
weights onto the laptop.
