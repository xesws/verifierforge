# v0.3.0 — Infrastructure: RunPod Runtime Provisioning

## Planned Change

Use the existing persistent `/workspace` volume for all provisioning state. The model cache root remains `HF_HOME=/workspace/hf-cache`; repository logs live at `/workspace/verifierforge/runs/provisioning/`. Run the long model downloads in tmux session `vf-downloads` and the dependency install in `vf-runtime-install` so closing the laptop SSH client cannot stop either operation.

## Persistent Paths and Data Flow

`huggingface_hub` writes downloaded model snapshots through `HF_HOME`; it must not use a cache outside `/workspace`. Set `PIP_CACHE_DIR=/workspace/pip-cache` for both the one-time download client bootstrap and trainer installation. Download and installation logs are append-only files under `/workspace/verifierforge/runs/provisioning/`, which are intentionally ignored by Git. The virtual environment stays at `/workspace/verifierforge/.venv` and is recreated by the existing idempotent bootstrap path if needed. The read-only deploy key's source of truth is `/workspace/.ssh/`; because this network volume cannot enforce private-key mode 0600, bootstrap creates a mode-0600 ephemeral copy in `/root/.ssh/` immediately before Git operations. No key material belongs in the repository or documentation.

## Detached Operations and Observability

Start sessions with explicit names and redirect stdout/stderr through `tee` to their persistent logs. Inspect with `tmux has-session`, `tmux capture-pane`, `tail`, and `nvidia-smi`; do not use foreground SSH as the worker lifetime. Existing `vf` commands are sufficient for D1 and need no speculative new control-plane abstraction in this version.

## Timeout, Failure, and Rollback Rules

The dependency work receives a hard 45-minute total timeout from its first session. The initial install was conclusively incompatible (CUDA 13 runtime dependency and a vLLM version outside verl's declared range), so preserve its venv and log, then allow one clean-venv compatibility correction only within the remaining time. On timeout or another unrecoverable resolver/CUDA error, terminate that session, preserve its log, add a concise failure record to `NOTES.md`, and leave already cached model files intact. Do not delete a working environment or model cache merely to retry. Download failures are best-effort and do not block the proven D1 loop.

For the workspace key, remove any stale repository-local `core.sshCommand` that points directly at the permissive network-volume file. The fallback is deterministic: bootstrap recreates the ephemeral secure copy from `/workspace/.ssh/`; no key is regenerated and no deploy-key permission changes are made.
