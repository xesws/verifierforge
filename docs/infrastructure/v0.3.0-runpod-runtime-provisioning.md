# v0.3.0 — Infrastructure: RunPod Runtime Provisioning

## Planned Change

Use the existing persistent `/workspace` volume for all provisioning state. The model cache root remains `HF_HOME=/workspace/hf-cache`; repository logs live at `/workspace/verifierforge/runs/provisioning/`. Run the long model downloads in tmux session `vf-downloads` and the dependency install in `vf-runtime-install` so closing the laptop SSH client cannot stop either operation.

## Persistent Paths and Data Flow

`huggingface_hub` writes downloaded model snapshots through `HF_HOME`; it must not use a cache outside `/workspace`. Set `PIP_CACHE_DIR=/workspace/pip-cache` for both the one-time download client bootstrap and trainer installation. Download and installation logs are append-only files under `/workspace/verifierforge/runs/provisioning/`, which are intentionally ignored by Git. The virtual environment stays at `/workspace/verifierforge/.venv` and is recreated by the existing idempotent bootstrap path if needed.

## Detached Operations and Observability

Start sessions with explicit names and redirect stdout/stderr through `tee` to their persistent logs. Inspect with `tmux has-session`, `tmux capture-pane`, `tail`, and `nvidia-smi`; do not use foreground SSH as the worker lifetime. Existing `vf` commands are sufficient for D1 and need no speculative new control-plane abstraction in this version.

## Timeout, Failure, and Rollback Rules

The dependency install receives a hard 45-minute timeout enforced on the pod. On timeout or an unrecoverable resolver/CUDA error, terminate that session, preserve its log, add a concise failure record to `NOTES.md`, and leave already cached model files intact. Do not delete a working environment or model cache merely to retry. Download failures are best-effort and do not block the proven D1 loop.

## Acceptance Evidence

Record pod Python/CUDA metadata, tmux session names, log paths, cache directory evidence, completion/error state, and the actual duration of the dependency attempt. Confirm all retained files live beneath `/workspace`.
