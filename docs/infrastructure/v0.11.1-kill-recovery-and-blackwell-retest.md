# v0.11.1 — Infrastructure: job-group cleanup and diagnostic capture

W1 changes the remote `vf` launch boundary so each job runs under `setsid` and
records its numeric PGID in `runs/<job>/pgid`. `vf kill <job>` first removes
the tmux session, sends TERM to that recorded process group, waits briefly,
sends KILL only if needed, then runs `ray stop --force` on this dedicated
single-job pod. It finishes by querying `nvidia-smi`; any remaining compute
allocation or nonzero GPU memory is a failure, not a silent success.

The acceptance exercise is an actual remote `fake_smoke` launch: detach, kill,
then assert no tmux session, no matching remote process, and zero GPU compute
memory. It is deliberately GPU-free so it validates lifecycle cleanup without
charging a model rollout. Logs and the PGID marker are returned through the
existing ignored `runs/` path.

W2 begins only after W1 passes. Its first fifteen minutes are read-only:
preserve raw `ray status`, the last 100 lines of driver and vLLM-engine logs,
GPU utilization/memory snapshots, and resolved GPU/TP/memory configuration.
The evidence is written into the run-sheet before any config change. It may
repair only a resource mismatch or a verified engine-init stall; two retests
or 45 minutes total exhaust the authorization.

**W1 result:** `w1-kill-v0111` was started with `fake_smoke`, which published
its PGID marker (`9991`) and 39 metrics records. `vf kill` completed after
removing tmux, signalling that group, and running the Ray fallback; it reported
zero GPU memory. A separate remote assertion confirmed no tmux session,
matching process, compute application, or GPU allocation remained. The synced
ignored evidence lives under `runs/w1-kill-v0111/`.

**W2 evidence result:** the diagnostics were captured under the ignored
`runs/p0-w2-diag-v0111/` path. Ray's default CLI could not discover the local
instance and an explicit GCS-address CLI probe timed out, while process/GPU
evidence independently showed a one-GPU raylet and a stalled WorkerDict. This
transport/CLI inconsistency is preserved as evidence; it is not treated as a
resource mismatch or repaired during the read-only window.

**W2 eager retest result:** W1 also cleaned up `p0-w2-eager-r1-v0111` after
its engine did not start; GPU memory returned to zero. The final retest must
make the attention backend visible to both the driver and Ray workers, rather
than relying on an SSH-session environment that would not survive worker
creation. It remains the final authorized W2 attempt.

**Final W2 stop:** the final runtime-env-propagated SDPA attempt never created
a GPU compute process before the W2 deadline. `vf kill` nevertheless reclaimed
the pod cleanly and reported 0 MiB. No further pod or control-plane mutation
is authorized under v0.11.1; the retained run directories are the only
evidence needed for the next human decision.
