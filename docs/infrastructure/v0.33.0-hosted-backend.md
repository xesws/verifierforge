## Hosted environment inventory

Railway sets `PORT`. Required application secrets/settings are:

- Access and database: `VF_REVIEW_INVITE_CODE`, `VF_DB_BACKEND=postgres`,
  `SUPABASE_DB_URL`, `VF_CRED_KEY`.
- Data and feature safety: `VF_API_DATA_MODE=hybrid`,
  `VF_AGENT_ENABLED=true`, `VF_AGENT_BINDING=mock`,
  `VF_AUTOPROVISION=false`, `VF_PROVISION_BINDING=mock`.
- Evidence: `VF_S3_BUCKET`, optional `VF_S3_PREFIX`, `VF_S3_REGION` or
  `AWS_DEFAULT_REGION`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, and
  optional `AWS_SESSION_TOKEN`.
- Proxy/Guardian: `VF_PROXY_UPSTREAM=fake`, `VF_PROXY_TUNED_UPSTREAM`,
  `VF_PROXY_TUNED_API_KEY`, `VF_PROXY_TUNED_MODEL` or `VF_ENDPOINT_MODEL`,
  optional Guardian sampling/window/pool variables.
- Browser: `VF_CORS_ORIGINS` for exact local/final origins and
  `VF_CORS_ORIGIN_REGEX=^https://[a-z0-9-]+\.vercel\.app$` for previews.

Do not set OpenAI, OpenRouter, or `RUNPOD_API_KEY` on the hosted reviewer.

## Railway deploy and rollback

The repository is deployable without GitHub repository authorization. The sole
existing project/service is linked and updated with:

```bash
railway link --project 1d930c52-79ea-49a8-8e4b-faa5057f5330 \
  --environment production --service verifierforge
railway up --service verifierforge --environment production --detach
railway domain update verifierforge-production.up.railway.app \
  --service verifierforge --port 8080
```

Before `railway up`, enter the variables in the inventory above through
Railway's secret manager. Do not pass secret values on a command line or copy
the local `.env` wholesale. The service command is supplied by
`scripts/start_hosted_backend.sh`; `railway.json` fixes `/healthz` and a single
replica. In the Railway dashboard, confirm the Hobby plan, disable Serverless,
and set a usage alert before sharing the generated `*.up.railway.app` domain.

Public acceptance is the frozen 19-operation matrix, Basic-auth rejection,
API/proxy CORS, one tuned completion, Guardian point creation, and degraded
fallback. Only after all pass may the old laptop reviewer and its Cloudflare
tunnel be stopped. Roll back application code with Railway's previous
deployment action. If inference alone fails, leave the application deployed
and update only `VF_PROXY_TUNED_UPSTREAM`; a restart is sufficient.

## H2/H3 result and endpoint rotation

The sole `invigorating-growth/verifierforge` service is live at
`https://verifierforge-production.up.railway.app`. Deployment
`dc6a6b5a-64e8-4afa-a735-cd005db547ab` initially exposed target port 8000 while
Railway injected `PORT=8080`; the container was healthy but the public edge
returned 502. Updating the existing domain target to 8080 fixed the route
without a rebuild or second service.

The public origin passed all 19 operations, Basic-auth rejection, API/proxy
CORS, the complete flagship report, twelve real tuned completions, and a new
Guardian point (129 → 130). The route was restored and all temporary Job,
decision, approval, credential, and S3 trace fixtures were removed.

The read-only H3 probe found the configured Cloudflare vLLM route healthy
(`GET /v1/models` HTTP 200, one model) while the RunPod-native fixed port URL
still returned HTTP 404. The old laptop reviewer and its public tunnel were
stopped only after H2 passed. A vfserve process audit found only vLLM and its
inference tunnel—no reviewer, API, or proxy process. Until port 8000 is
explicitly exposed by RunPod or a fixed free provider URL is available, tunnel
rotation is intentionally one variable:

```text
Railway VF_PROXY_TUNED_UPSTREAM=https://<new-tunnel>/v1
```

If that URL expires first, `/healthz` becomes `degraded`, tuned proxy requests
fall back to the deterministic default, and flagship reports remain complete
from artifacts/S3 rather than returning a red error page.
