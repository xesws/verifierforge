# v0.37.0 Railway/Vercel demo traffic rollout

The singleton guarantee depends on the committed Railway topology: one replica
and one Uvicorn worker. `VF_DEMO_TRAFFIC_ENABLED` defaults false. The owner has
authorized enabling it for this production rollout after the flag-off backend
smoke passes; increasing replica or worker count requires disabling the feature
until a distributed lease exists.

Deploy the tested backend SHA to Railway first, verify health plus authenticated
and anonymous demo responses, then enable the flag and deploy the matching
frontend to the linked `verifierforge-web` Vercel project. Do not add any legacy
static tuned-endpoint variables; the proxy continues to resolve only the ready
Supabase registry record.

Live acceptance records traffic and Guardian high-water marks, performs the
cold run first, then one explicitly confirmed Wake and ready 50% SQL canary run.
Use the established three-minute acceptance idle override, restore the normal
30-minute policy and original routing afterward, and require registry cold plus
provider inventory zero. Rollback order is flag off, prior Vercel production,
then prior Railway deployment if the API itself regressed.
