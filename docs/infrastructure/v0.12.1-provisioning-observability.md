# v0.12.1 — Infrastructure: truthful bootstrap completion

N3 D1 evidence was definitive: `tmux ls` returned `error connecting to
/tmp/tmux-0/default (No such file or directory)` and
`/workspace/verifierforge/runs/provisioning/` was absent, including any
`runtime-install` log. The H100 volume therefore had no in-flight installation
to wait for.

The repair makes installation durable and inspectable. `vf bootstrap` prepares
the clone/key preconditions, launches a named `vf-bootstrap` tmux session, and
waits for its status artifact. The session records its stdout/stderr in
`runs/provisioning/runtime-install.log`; an EXIT trap writes a timestamped
success/failure status. A waiting caller returns success only for a zero status.
If the caller loses SSH, the session continues, but a later invocation must
observe the status rather than report a false completion.

The new H100 key remains copied from the permissive workspace volume to a
`0600` root runtime copy before Git operations. The repair does not loosen
host-key, deploy-key, version, or artifact checks.
