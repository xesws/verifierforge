# v0.11.0 — Infrastructure: D4 Blackwell migration and persistence

The former worker was a disposable single NVIDIA L4. M1′ replaces it with one
NVIDIA RTX PRO 6000 Blackwell Server Edition (96 GB) while preserving the
`/workspace` network volume. Step 0 verified `verifierforge`, `hf-cache`, and
`pip-cache` on that mounted volume before bootstrap. `vf bootstrap` remains
idempotent and must restore the repository, venv, HF cache, and remote
preflight without copying laptop `.env` or any API key.

`vf train` continues to detach through tmux without attempting unsafe concurrent
GPU sharing. The control runs after the main job on the same one-GPU worker.
`vf watch` remains the
only laptop return path for metrics/logs/selected small artifacts; it excludes
checkpoints and weights. The main and control jobs need separate tmux names and
Storage roots so their curves cannot overwrite each other.

For the first 30 minutes, readiness for AWS is checked by variable/credential
availability only and never printed. If a tested S3 backend is not ready, the
main run explicitly uses LocalStorage on the network volume; its later
Gate-B-scale S3 proof is a non-blocking follow-up. Lifecycle operations are
operator actions rather than a new automatic RunPod API layer. Lack of a
credential or tool to stop/provision pods is a hard M1 stop before any training.
