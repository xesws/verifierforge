# v0.16.1 native checkpoint recovery — infrastructure

**Status:** implemented and locally validated; pod proof pending.

The H100 S7 job stopped at step 50 before checkpoint publication. Exact cause:
the strict serving gate reported `no safetensors files in .../actor/huggingface`.
The native payload itself is complete (`model`, `optim`, `extra_state`, and
`data.pt`) and is preserved in the run cache's failed-staging evidence.

The recovery tool uses a manifest-last S3 checkpoint wrapper, not a filesystem
copy or a trainer patch. Credential values enter only a short-lived tmux
environment from stdin; the lifecycle evidence records `injected` and
`cleared`, never a value. The worker resumes solely from the S3 materialization.

The recovery command will preserve the first attempt's `train.log` as an S3
artifact before the resume invocation overwrites its local tee destination.
