# v0.34.0 Scale-to-zero serving infrastructure

**Status:** complete; locked fallback accepted in two live cycles

The hosted reviewer remains the control plane. It receives only platform-held
RunPod, S3, Fernet, and invitation credentials. No
secret is written to the repository, command output, audit detail, or pod disk.
The pod receives short-lived presigned S3 URLs rather than AWS credentials.

Runtime defaults: one session, `$5` session budget, 120-minute maximum runtime,
30-minute idle timeout, vLLM 0.10.2, model length 2048, bf16, and served model
`vf-demo`. Cloudflared is pinned to `2026.7.2` with linux-amd64 SHA-256
`ec905ea7b7e327ff8abdde8cb64697a2152de74dbcdbf6aec9db8364eb3886cd`.

The preferred runtime is the official preinstalled vLLM image pinned by digest.
If the 45-minute acceleration timebox fails, the verified PyTorch image plus
locked install remains the documented fallback and does not weaken identity or
readiness gates. Existing L4 processes remain untouched until owner action.

No AWS credential enters the pod. The controller creates a short-lived S3
download manifest whose entries are themselves presigned; the pod verifies all
file hashes and the M6 canonical tree before starting vLLM. A presigned S3 PUT
returns the quick-tunnel URL without SSH or a callback credential. The public
endpoint key is stored only as Fernet ciphertext.

SV4 live result: the official digest allocated an L4, but RunPod exposed no
container runtime, port mapping, or S3 callback during the 30-minute readiness
window. The fuse deleted it and provider inventory returned zero. This path is
therefore not the selected runtime. The bounded fallback uses
`runpod/pytorch:1.0.2-cu1281-torch280-ubuntu2404` and installs exactly
`vllm==0.10.2` before the unchanged identity/readiness gates.

The fallback bootstrap additionally treats readiness evidence as a small state
machine: `tunnel_published`, `vllm_starting`, `ready`, or `failed`. A failed
vLLM child overwrites the presigned callback with a redacted, bounded diagnostic
record. The control plane recognizes `failed` and drains immediately. This is
the production diagnostic path; SSH is not a correctness dependency and the
pod never receives a reusable private key.

The first evidence-bearing fallback attempt exited with code 1 at tokenizer
initialization: `Qwen2Tokenizer has no attribute all_special_tokens_extended`.
This is the same dependency class already closed by v0.15.2. The selected
fallback must install the complete proven serving quartet from
`requirements-serve.txt`: vLLM 0.10.2, Transformers 4.57.6, tokenizers 0.22.2,
and huggingface-hub 0.36.2. Pinning only vLLM is not a reproducible runtime.

The accepted fallback cold starts were 282.14 and 266.68 seconds on RTX 4000
Ada at `$0.20/hr`. The 3-minute acceptance idle override reclaimed each pod;
both DELETE readbacks found the target absent and the full managed inventory at
zero. Production retains the 30-minute idle default. Billing evidence remains
the established three-part policy: DELETE and inventory-zero are synchronous,
while provider billing rows are collected asynchronously at +1h and +6h.
