# v0.15.2 — L4 runtime compatibility recovery

The disposable `vfserve` L4 pod retains vLLM `0.10.2` and torch unchanged.
Only `transformers` and `tokenizers` are aligned to the training-pod freeze
after S1 captures all three machines' inventories. Each retry starts from a
stopped tmux service, keeps warning-level vLLM logging, scans logs for secret
markers before display, and tests loopback `/v1/models` plus one completion.

At most three minimal alignments are allowed. A fourth attempt, a new model,
or any hidden package upgrade is out of scope. Failure becomes an evidence
record and S5--S6 are skipped; it does not block storage or submission work.

**S1 evidence:** the training runtime is `transformers==4.57.6` and
`tokenizers==0.22.2`, while the failing serving runtime is
`transformers==5.14.1` and `tokenizers==0.22.2`. The only material dependency
drift implicated by the tokenizer error is therefore `transformers`; all three
complete freezes are retained under the ignored M6 evidence directory and
their SHA-256 values are recorded in the run sheet.

**Service result:** the first direct dependency install aligned Transformers
and tokenizers; `pip check` then exposed only `huggingface_hub==1.23.0` as
outside Transformers 4.57.6's `<1.0` range. Aligning that one package to
`0.36.2` passed `pip check`, loaded the L4 model, and produced a local real
completion. The configured public RunPod proxy hostname did not return any
bytes within 30 seconds; this is an exposure/routing blocker, not a model-load
failure. The detached local service remains up and key-clean.
