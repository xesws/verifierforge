# v0.28.2 — RunPod billing reconciliation

Every P-2 deletion writes a durable schedule with deletion time and two due
slots: `+1h` and `+6h`. The reconciliation command attempts each slot at most
once, writes atomically, and appends a provider billing record when one exists.
An empty response remains pending and never reopens a safely terminated pod.

The historical gold deletion audit is `2026-07-19T07:56:44Z`, so its slots are
`2026-07-19T08:56:44Z` and `2026-07-19T13:56:44Z`. New orphan and training slots
are derived from their actual deletion timestamps. Evidence contains no API
keys, Authorization headers, SSH endpoints or metadata for unrelated pods.

On 2026-07-19, two Community create attempts allocated nothing and returned the
same four-line `There are no instances currently available` error. Official
read-only stock data showed Secure low stock for the already-approved RTX 4000
Ada and L4 IDs at `$0.26/hr` and `$0.39/hr`; v0.28.2 permits that same-list
Secure retry without changing disk, persistence, interruptibility or budget.

Secure REST responses may omit `runtime` and use a Go-style UTC suffix in
`lastStartedAt`. Uptime estimation therefore falls back to a normalized
provider timestamp, with a regression test for fractional seconds and
`+0000 UTC`. The live run also uses an external wall-clock guard so the parser
repair does not require interrupting healthy training.

The first due reconciliation after the live attempts found no orphan billing
row and confirmed `$0.124736` for the original full attempt. Seven detached,
idempotent local tmux sessions now cover the remaining per-schedule `+1h` and
`+6h` slots; their logs and atomic schedule updates remain under ignored
`runs/provisioning/v0.28.2/`. Pending rows do not alter deletion evidence.
