# v0.28.2 — RunPod billing reconciliation

Every P-2 deletion writes a durable schedule with deletion time and two due
slots: `+1h` and `+6h`. The reconciliation command attempts each slot at most
once, writes atomically, and appends a provider billing record when one exists.
An empty response remains pending and never reopens a safely terminated pod.

The historical gold deletion audit is `2026-07-19T07:56:44Z`, so its slots are
`2026-07-19T08:56:44Z` and `2026-07-19T13:56:44Z`. New orphan and training slots
are derived from their actual deletion timestamps. Evidence contains no API
keys, Authorization headers, SSH endpoints or metadata for unrelated pods.
