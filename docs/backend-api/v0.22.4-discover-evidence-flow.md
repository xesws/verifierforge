# v0.22.4 — Discover API parity and approval receipt

The true API gains the cluster-list/detail routes already exposed by the mock,
using the existing `Cluster` response contract. Stable profile values live in a
shared catalog; for Data Pull SQL they are `monthly_calls=95_000` and
`monthly_cost_usd=5500.0`.

Agent Analyze remains backward compatible with a bodyless POST and additionally
accepts a small app-local `data_source` request. A provided path must resolve to
the configured `VF_PROXY_DB_PATH`; mismatches fail with 422 before any Agent or
LLM work. The Discover HTML receives the configured default without exposing a
secret.

`GET /agent-decisions/{decision_id}/approval` returns the existing
`ApprovalRecord` or 404. Decision and approval reads use store-only services so
they do not construct an LLM client, S3 trace store, provisioner, or trainer.

## Validation result

Contract parity tests compare the true and mock cluster payloads. Data Pull SQL
is exactly `95_000` calls and `$5_500`; configured input succeeds, a different
path fails with 422 before Agent construction, and the legacy bodyless Analyze
request remains valid. Approval POST/GET idempotency and reload recovery are
covered by API tests.
