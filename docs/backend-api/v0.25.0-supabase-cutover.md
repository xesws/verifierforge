# v0.25.0 — Supabase cutover proof

The migration is a single import, not dual write. Legacy rows receive stable
identities derived from a fixed source namespace and legacy primary key, so a
second execution is a no-op. Agent decision summaries retain all current audit
fields; approvals remain unique by decision; raw metrics and traces remain in
Storage/S3.

Postgres tests bind repositories to a rollback-owned connection. The real API
and mock contract-shape tests remain unchanged. The owner authorized the
persistent `.env` switch only after migration, reconciliation, and a
process-local smoke succeeded.

The direct hostname failed DNS from the IPv4-only development host; the
Supavisor session pooler then passed. The product smoke proved proxy traffic,
clusters, routing, guardian points, Discover, Agent decisions, and approvals
against Supabase. A cutover regression also made caller-supplied Agent SQLite
paths authoritative over the global Postgres selection. Final validation was
`365 passed, 1 skipped` with Postgres selected.
