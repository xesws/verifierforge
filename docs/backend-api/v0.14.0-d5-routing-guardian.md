# v0.14.0 — Routing control API

The real API adds `GET` and `PUT /clusters/{cluster_id}/routing`. Responses and
updates use the existing `RoutingState` model exactly:
`cluster_id`, `enabled`, `canary_percent`, and `target_model`. Internally,
`target_model` maps to the proxy SQLite table's `target_upstream` column so the
public contract remains stable while routing stays transport-oriented.

The API also serves `GET /clusters/{cluster_id}/live-pass-rate` as the existing
`LivePassRate` shape. The mock exposes the same endpoints and model-validated
shapes, enabling the frontend to use either service without a special case.

All state is local SQLite under the configured proxy database path. Missing
routes read as a disabled, zero-canary state for the requested cluster. Route
updates validate the 0–100 bound through Pydantic; route or guardian database
errors are isolated from the proxy's customer-facing completion path.

Local fake-mode acceptance PUT `data-pull-sql` to 50%, then verified that the
real API returned the stored state and a 113-point `LivePassRate` series with
the exact shared field names. A subsequent PUT to 0% produced no new tuned
route records.
