# v0.34.0 Serving wake API

**Status:** planned

Add `POST /serving/wake` and `GET /serving/status`. Wake requires the reviewer
invitation boundary, `confirm_provider_spend=true`, and
`VF_SERVING_WAKE_ENABLED=true`; the code default is false. A new cold request
returns HTTP 202, while an idempotent request for an existing provisioning,
loading, or ready session returns HTTP 200 without another provider mutation.

Responses expose only a sanitized `ServingStatus`: session/model identity,
state, optional ready URL, progress/error code, selected GPU and public cost
facts. The vLLM key and its database reference are never serialized. When the
registry is cold or unavailable, report and arena endpoints continue to serve
immutable artifact data.

Validation covers flag-off behavior, Basic Auth, CORS, idempotence, database
failure, no-capacity, sanitized errors, real/mock OpenAPI parity, and polling
across every serving state.
