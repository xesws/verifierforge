# v0.22.2 — OpenAI Responses transport

OpenRouter remains on Chat Completions. OpenAI transport selection will be made
only from the recorded three-probe matrix and implemented behind the existing
generic `LLMClient` surface.

`LLMRequestError` must expose the HTTP status and a bounded provider response
body without API keys or request headers. Gate C must run one plain and one tool
preflight request before any scenario batch.

The live matrix selected Responses: plain Chat returned `OK`, Chat function
calling returned HTTP 400, and Responses returned a completed
`function_call`. The implementation follows the official
[function-calling flow](https://developers.openai.com/api/docs/guides/function-calling#function-tool-example):
Responses tools use the flattened function shape, and each tool output is sent
as `function_call_output` correlated by `call_id`. `previous_response_id` keeps
the server-side reasoning items available across turns.

OpenRouter's request and parsing path is unchanged. The Gate C preflight now
uses the same provider-selected production transport as the batch.
