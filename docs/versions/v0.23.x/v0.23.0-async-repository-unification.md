# v0.23.0 — Async repository unification (DB-1)

**Status:** complete — 2026-07-18
**Owner/worktree:** Codex integration owner on `main`
**Release type:** minor database architecture capability

## Problem and scope

Traffic, routing, guardian points, Agent decisions, and approvals currently
open SQLite directly from multiple product modules and create schema during
reads. DB-1 replaces those paths with one SQLAlchemy 2.0 async repository layer
and an Alembic-managed schema while preserving SQLite as the default local
backend. It also defines Cluster, Job, Credential, and Provision audit stores
needed by DB-2 and P-1.

`VF_DB_BACKEND=sqlite` remains the default and resolves `VF_PROXY_DB_PATH` to
an `aiosqlite` URL. `VF_DB_BACKEND=postgres` requires `SUPABASE_DB_URL` and
normalizes it to `postgresql+asyncpg`; it never falls back or logs a URL.

## Non-goals

Do not run a true Supabase migration, change `.env`, encrypt credentials, call
any provider, alter trainer/frozen data, change response fields, or make the
Provisioner executable. S3 remains the large-object store; relational stores
hold only queryable facts and bounded job summaries.

## Affected areas and contracts

- Async protocols: `TrafficStore`, `ClusterStore`, `RoutingStore`,
  `LivePassRateStore`, `JobStore`, `AgentDecisionStore`, `CredentialStore`,
  `ApprovalStore`, plus internal `ProvisionAuditStore`.
- SQLAlchemy models and Alembic schema for the eight design tables, with
  `guardian_scores` and `provision_events` support tables.
- Proxy/API/Agent/tools/summary scripts depend only on injected stores; no
  product module imports or catches `sqlite3`.
- Existing synchronous product seams use one process-local gateway whose async
  engine stays on a dedicated event loop; callers do not create per-request
  pools or bypass the repository interfaces.
- Existing HTTP/Pydantic response shapes, artifact reviewer mode, Agent flags,
  Storage ABC, and mock server remain compatible.

## Validation plan

Run the repository contract suite against a temporary migrated SQLite DB;
exercise Alembic upgrade/downgrade/upgrade; keep the existing mock/true API
shape assertions; prove proxy accounting/guardian failures do not break a
completion; and run the complete suite. A production-code grep may match
`sqlite3` only in the in-memory NL2SQL verifier and the future one-shot legacy
importer. Passing all gates creates `db-1-complete`.

## Result

Alembic owns ten relational tables, including the two internal support tables.
An unmanaged legacy SQLite file is rejected before DDL runs, so DB-2 can import
it without risking a partial in-place migration. Repository, gateway, API,
proxy, Agent, migration, and full-suite validation passed; production grep has
no direct `sqlite3` outside the in-memory verifier and DB-2 importer.
