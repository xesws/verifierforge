# v0.39.1 — Leave session returns serving to cold

**Status:** in progress

## Problem and scope

The Reviewer `Leave session` action currently clears browser state only. A
ready scale-to-zero endpoint can therefore remain allocated after the reviewer
leaves, which violates both the demo reset expectation and the cost boundary.

This patch adds an invitation-protected, idempotent serving sleep operation and
changes Leave session to wait for confirmed provider deletion and a durable
`cold` registry state before clearing the invitation and journey.

## Explicit non-goals

- No change to Wake, training, frozen evidence, routing, Guardian, or trainer.
- No browser-unload or tab-close shutdown: browsers cannot reliably complete a
  provider deletion request during page teardown.
- No hidden best-effort logout that reports success while a GPU may remain.

## Affected areas and ownership

- `core/` and `app/api/`: additive sleep request and serving endpoint.
- `app/serving/`: serialized, idempotent drain entrypoint.
- `frontend/`: Leave session shutdown state, success, and retryable failure.
- `docs/backend-api/`, `docs/frontend/`, and `docs/infrastructure/`.
- Owner/branch: primary Codex agent on
  `fix/v0.39.1-leave-session-cold`; no parallel worktree.

## Contract impact

Additive only: `POST /serving/sleep` accepts `{ "model_id": "vf-demo" }` and
returns the existing `ServingStatus` contract. Authentication remains the
reviewer invitation. A cold or absent session returns cold without provider
work; an active session returns only after provider deletion is confirmed.

## Validation plan

1. Contract and API tests cover authentication, active shutdown, already-cold
   idempotency, unknown model, and sanitized failure responses.
2. Coordinator tests cover one provider termination, concurrent serialization,
   and durable cold state.
3. Frontend tests cover the waiting label, successful reset, and a failure that
   preserves the authenticated session for retry.
4. Run frontend tests/build plus full `pytest -q`; confirm no secret or unrelated
   runtime file is staged.
