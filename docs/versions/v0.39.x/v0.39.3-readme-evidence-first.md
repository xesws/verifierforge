## Problem and scope

The v0.39.2 README explains the business and reviewer path, but the first
screen still makes judges read setup and architecture before seeing the actual
NL→SQL example and the central quality result. Reorder the repository landing
page around the shipped scenario and its evidence.

The front of the README will show one frozen-schema business question, the
kind of SQL the specialist produces, and the canonical held-out checkpoint
figure proving pass@1 moved from 58.3% to 78.3% on 60 unseen prompts. It will
also surface the verifier pipeline and the main/random-reward monitoring figure
before infrastructure details. Contributor deployment and database operations
will be compressed behind a secondary details section or linked documentation.

## Explicit non-goals

- No new figure, metric, training claim, evaluation, model call, or deployment.
- No API, frontend, backend, trainer, verifier, dataset, or contract change.
- No claim that the M3/M4 curves form a strict single-variable experiment.
- No secret, invitation code, model weight, or runtime artifact publication.

## Affected areas and contract impact

- `README.md`: information order, canonical figure placement, concrete example,
  and reduced contributor/operations prominence.
- `docs/submission/`: judge-facing message and evidence provenance.

There is no contract impact. All figures are existing deterministic SVGs built
from committed evidence and guarded by `tests/test_blog_assets.py`.

## Validation plan

- Verify the example against the frozen schema and reviewed Ship prompt set.
- Verify the 58.3%/78.3%/60-prompt claims against the held-out report.
- Run canonical blog-asset checks, secret scanning, link checks, and the full
  repository test suite.

  checkpoints, baseline pass@1 `0.5833`, selected step 350 pass@1 `0.7833`,
  and step 400 pass@1 `0.7167`.
- Canonical blog assets: 17 current; focused tests: 8 passed; secret scan:
  passed; full suite: 497 passed, 1 credential-gated skip.
