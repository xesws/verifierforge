# v0.39.0 — live Guardian auto-traffic

**Status:** validated; deployment pending

## Problem and scope

Saving the Ship routing policy currently updates only the policy. Reviewers
must separately start demo traffic, and the Guardian chart renders the entire
historical point set, so newly sampled scores occupy only a few pixels and the
chart does not visibly advance with the 200-request run.

After a successful `Save routing policy`, automatically adopt or start the
existing 200-request demo task. During that task, render a run-local Guardian
curve whose x-axis advances from request 0 through request 200 as `sent`
progress arrives. Each traffic moment carries forward the latest observed
rolling Guardian pass rate; only a real newly persisted Guardian score may
change the y-value. Preserve the completed run-local curve and summarize its
request and verifier-sample deltas after the task stops.

## Explicit non-goals

- No claim that all 200 requests are verifier samples. Guardian sampling stays
  limited to successful tuned Data Pull SQL responses.
- No fake pass-rate values, interpolated verifier outcomes, or hidden model
  reasoning.
- No API, backend task, database, routing, canary, serving, or sampling-policy
  change.
- No automatic Wake and no change to cold/default-fallback behavior.

## Affected areas and ownership

- `frontend/src/pages/ShipPage.tsx`: save-to-traffic handoff and run-local
  traffic/Guardian observation state.
- `frontend/src/components/GuardianChart.tsx`: dynamic 0..200 run curve and
  honest live/completed summary.
- `frontend/src/components/guardianRun.ts`: pure, testable expansion from the
  observed `sent` counter to one carry-forward chart point per request.
- `frontend/src/components/DemoTrafficControl.tsx` and frontend styles/tests as
  required for durable completion feedback.
- `docs/frontend/v0.39.0-live-guardian-auto-traffic.md`: matching UI behavior.

Owner/worktree: primary Codex agent on `main` under the trunk-first rule.
Contract impact: none; the existing additive `/demo/*` extension is reused.

## Validation plan

- Prove a successful routing save immediately POSTs `/demo/traffic` once and
  adopts an already-running task without duplicate local starts.
- Prove progress expands into exactly one run point per `sent` request, up to
  200, while y-values come only from the latest API-returned Guardian value.
- Prove the chart updates before completion and retains its final run view and
  summary after completion.
- Preserve the manual button, cold guidance, request concurrency guard, and
  normal historical chart before a run.
- Run focused frontend tests, the full frontend test/lint/build gates, and the
  repository pytest suite before commit and deployment.

## Validation result

The implementation passed 34 frontend tests, ESLint, the production Vite
build, `git diff --check`, `bash -n scripts/vf`, and the repository suite with
492 passed and 1 skipped. The repository suite emitted its existing
`aiosqlite` event-loop-close thread warning but had no test failure.

The available browser runtime reported no browser instances, so no visual
animation claim or screenshot is recorded. The run-projection unit test proves
request 1 through request 200 each receive one chart point, and the page-flow
test proves routing PUT precedes exactly one demo-traffic POST and exposes the
live run state before completion.
