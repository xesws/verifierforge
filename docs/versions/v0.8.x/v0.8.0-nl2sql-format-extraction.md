# v0.8.0 — NL2SQL fenced-SQL extraction

**Status:** A1 implemented; A2 passed; A3 pending
**Owner / worktree:** Codex / repository-root `main`
**Predecessor:** `v0.7.1` auditable format-failure routing (Branch A)

## Problem and scope

The immutable subset evidence established that 322/341 failed model completions
are complete SQL statements wrapped only in Markdown code fences. v0.8.0
implements Branch A's extraction normalization before the existing tiered
scorer: it strips a SQL/untagged Markdown fence, uses `sqlparse` to select the
first statement in that extracted block, then sends that text through the
unchanged parse/read-only/SQLite/result/length scoring path. The verifier is
identified as version `2` (the legacy implementation is version `1` by source
identity).

v0.8.0 also adds a deterministic offline A2 re-verification command and
evidence file for all 276 stored candidates. It must complete with exactly 276
full passes before any fresh local-vLLM Gate A request is allowed.

## Explicit non-goals

- No change to tier thresholds, SQLite safety authorizer, expected-result
  comparison, length penalty, model, trainer logic/configuration, candidate
  data, projection, freeze, or tag.
- No OpenRouter request, `.env` use, or external model call.
- No normalization of unfenced prose or unfenced multi-statement SQL. This is
  an explicit safety judgment: it prevents a trailing raw DML statement from
  being silently discarded. The documented first-statement rule applies inside
  a recognized Markdown SQL/untagged code block.

## Affected areas

- `core/rewards/nl2sql.py`: versioned extraction wrapper plus unchanged raw
  tier scorer.
- `core/eval_runner.py` and `scripts/gate_a.py`: additive verifier/extraction
  facts in sample and aggregate evidence.
- `scripts/reverify_nl2sql_dataset.py`: A2's atomic, offline candidate
  re-verification evidence.
- `tests/`: fenced extraction, safety boundary, version/evidence, reverify,
  and atomic-publication coverage.
- `docs/verifiers/`, `docs/evaluation-serving/`, `docs/infrastructure/`, and
  `docs/p0-run-sheet.md`.

## Acceptance and stop conditions

A fenced exact query must obtain the same tier scores as its inner SQL; raw
unfenced multi-statement SQL must still fail the single-statement gate. A2 must
bind its output to all 276 input records, verifier v2, and source hash; any
score below `1.0` is stop condition ③. Only an A2 success permits the Branch A
subset Gate A rerun, whose fixed threshold remains the sole admission decision.

## A1 validation result

The implementation exposes `NL2SQLVerifier.VERSION == 2`, retains raw model
completion evidence, and adds the scored completion/extraction facts without
removing prior evidence fields. Tests cover fenced exact SQL, first statement
selection inside a fence, unfenced multi-statement rejection, v2 evidence, and
atomic re-verification publication. The full suite completed with `154 passed,
1 skipped`; A2 remains the required offline 276-row acceptance gate.
