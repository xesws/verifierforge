# v0.35.4 — Live SQL execution

Status: deployed; owner visual acceptance pending

## Problem and scope

The Ship playground currently proves that the tuned model generated SQL, but
does not execute that SQL or show database rows. This patch adds an explicit,
second reviewer action that runs the exact generated text against the frozen
NL2SQL demo dataset and displays the real result or SQLite error.

The preferred implementation is `sql.js`/WASM in a Web Worker. The worker uses
the repository's frozen schema and synthetic INSERT fixture, creates a fresh
in-memory database for every run, and never looks up a prompt's reference SQL
or expected result. Browser integration has a 30-minute feasibility gate. If
WASM loading, the worker smoke, and the production Vite build do not all pass
inside that window, implementation switches to an invite-protected ephemeral
server endpoint with the same read-only and timeout guarantees.

## Non-goals

- Connecting to a customer, Supabase, or production database.
- Automatically executing SQL as soon as generation finishes.
- Claiming semantic correctness or a verifier score for arbitrary questions.
- Changing model serving, routing, provisioning, training, or frozen data.
- Streaming precomputed rows or inventing execution progress.

## Contract impact

The preferred browser implementation adds no public HTTP operation, so the
frozen frontend contract remains 22 operations. Its internal worker result is
typed and includes a unique execution ID, completion ID, normalized SQL and
hash, real columns/rows, row count, truncation, duration, timestamp, SQLite
version, and frozen dataset identity.

Only if the 30-minute browser gate fails may the fallback add operation 23,
`POST /serving/sql-executions`, with real/mock OpenAPI parity.

## Validation plan

- Deterministic exporter/check proves the browser fixture matches the canonical
  source SHA and unique schema SHA.
- Real `sql.js` tests cover SELECT/WITH, fenced SQL, NULL/zero rows, truncation,
  SQLite errors, writes, multiple statements, and a terminated long query.
- Two executions containing `randomblob()` must return different values,
  execution IDs, and timestamps.
- Ship tests cover the separate Generate/Run actions, real activity states,
  stale-result clearing, duplicate-submit protection, cold-model independence,
  error rendering, keyboard use, and responsive tables.
- `npm test`, `npm run lint`, `npm run build`, and `pytest -q` must pass before
  push or deployment evidence is recorded.

## Implementation result

The browser path passed its feasibility gate in under ten minutes, so the
server fallback and operation 23 were not implemented. Vite emitted a dedicated
48KB worker and 660KB WASM asset. Real `sql.js` tests executed the frozen rows,
surfaced native missing-column errors, capped 150 generated rows at 100, and
returned different `randomblob()` values for repeated executions.

Validation on 2026-07-20:

- frontend: 25 tests passed across five files;
- ESLint: clean;
- TypeScript/Vite production build: passed;
- repository: 479 passed, 1 skipped;
- frozen asset check: source and schema identities matched.

The in-app browser discovery returned no available browser instance, so no
screenshot is claimed from this session. Public deployment and owner visual
acceptance remain separate evidence steps; the missing browser surface does not
change the passing real-runtime, Worker bundle, or build results.

Vercel production deployment `dpl_4RbQL5VWB99WJgRw7FH3fs5pQXdm` reached READY
and was aliased to `https://verifierforge-web.vercel.app`. The public Ship deep
link, 48KB Worker, and 660KB WASM each returned HTTP 200; the WASM response used
`application/wasm`, and the served Worker contains the frozen schema SHA and
dataset ID.
