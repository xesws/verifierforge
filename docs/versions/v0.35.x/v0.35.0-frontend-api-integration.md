`feature/v0.35.0-frontend-api-integration`

## Problem and scope

Nora's `feature/v0.18.0-static-frontend` is the accepted visual base, but it
does not call the frozen frontend API. This feature preserves that visual
language while wiring the `frontend-api-v1` operations, runtime invitation
authentication, explicit loading/error/cold states, the separate
Analyze → Approve → Start Forge boundaries, scale-to-zero serving status, and
Vercel delivery. Owner review found that the first integrated layout collapsed
discovery, authorization, proof, and serving into one screen. The acceptance
correction therefore enforces the product dependency
`Discover → Forge → Runs → Proof → Ship`, moves every serving action to Ship,
and makes the ready-only SQL probe call the registered tuned endpoint directly.

The integration starts from current `main` and cherry-picks Nora's exact
`642e3354457529d0c51d65a10bb2e2b4b36008ec` commit so authorship remains
visible. One budget-fused real serving wake is authorized for acceptance; no
Forge training is authorized.

## Explicit non-goals

- No trainer, frozen-data, provisioner lifecycle, or database schema changes.
- No raw pod-log transport or fabricated download/checkpoint progress.
- No change to production canary policy while a reviewer probes the tuned model.
- No redesign or replacement component library.
- No real Start Forge action while `VF_AUTOPROVISION=false`.
- No invite code, provider credential, database URL, or cloud key in source,
  build variables, screenshots, recordings, logs, or commits.

## Contract impact

Additive. The frontend boundary grows from 21 to 22 operations with
`POST /serving/tuned-completion`. It accepts an OpenAI-compatible non-streaming
chat request, returns the upstream completion unchanged, and exposes
`X-VerifierForge-Route: tuned`. A cold endpoint is HTTP 409 and an upstream
failure is a secret-safe HTTP 502; this path never consults or mutates canary.

## Stop conditions

Do not merge or tag on a contract mismatch, leaked credential, unproven GPU
deletion, failed production deep link, unresolved CORS failure, or missing
owner visual acceptance. A blocked independent item is recorded after 30
minutes and skipped; it does not silently become accepted.

## Implementation checkpoint

F2/F3 removed the build-time demo-artifact generator and all browser-local Job
fixtures. Discover, Forge, Job, Report, and Ship now consume only the typed API
client. Fourteen frontend tests pass, followed by a warning-free ESLint run and
a production TypeScript/Vite build; the remaining Vite chunk-size advisory is
a performance follow-up, not a correctness failure. The repository-wide suite
passes with `478 passed, 1 skipped`.

F5 deployed `https://verifierforge-web.vercel.app`, proved the five direct
product deep links, verified the built Railway origin, and received an exact
Vercel-origin CORS response from Railway. The public entry is live on the
feature branch; it is not yet the owner-approved/tagged F6 release.

### Browser compatibility defect

The first owner browser check exposed `Failed to execute 'fetch' on 'Window':
Illegal invocation`. The typed client had retained native `fetch` and later
invoked it as a client-object method, supplying an invalid receiver in WebKit.
Before further acceptance or Wake activity, bind the default transport to
`window`, add a native-receiver regression test, rebuild, and redeploy the same
v0.35.0 candidate. This is a pre-release correction, not a released patch.

### Reviewer journey acceptance defect

Manual review then found that Discover still contained approval, Start Forge,
historical proof, and Wake controls, while every navigation item and deep link
was immediately reachable. The correction stores only the current invitation
session's progress, validates its decision/approval/job against the API, and
fails closed to the earliest unmet stage. Existing shared Supabase facts never
auto-unlock a fresh reviewer.

The public sandbox keeps `VF_AUTOPROVISION=false`, so an approved reviewer may
explicitly select the cluster's already-completed flagship job to continue the
evidence walkthrough. This is labelled as historical evidence and never
pretends that a new training run started.
