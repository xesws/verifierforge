# v0.35.3 — Ship readiness messaging

**Status:** complete on deployed feature candidate

**Date:** 2026-07-20

**Owner / worktree:** Codex / repository root branch
`feature/v0.35.0-frontend-api-integration`

## Problem and scope

The tuned SQL button looked active while serving was cold or loading, yet used
a native disabled state and produced no click feedback. The page did not state
the required `Wake → Loading → Ready → Run` dependency next to the action.
Separately, registry polling paused in a background tab but did not resume on
visibility restoration, leaving a genuinely ready endpoint displayed as
loading.

Make readiness a visible product gate, never a silent code guard. Add
state-specific instructions and button labels, make a premature Run click emit
an explicit local warning and focus the Wake section, give disabled controls a
real disabled appearance, and resume polling when the tab becomes visible.

## Non-goals

- No automatic Wake, GPU allocation, tuned request, or canary mutation.
- No API, database, serving runtime, verifier, model, or trainer change.
- No invented progress percentage or raw pod-log claim.

## Affected areas

- `frontend/src/pages/ShipPage.tsx`: readiness gate and non-silent action.
- `frontend/src/components/WakeModelControl.tsx`: stable Wake anchor.

- `frontend/src/styles/index.css`: warning/success gate and disabled visuals.
- frontend tests, frontend documentation, and run-sheet.
