# v0.4.0 — D2 GRPO Smoke, Verifier Copilot, and Sandbox

**Status:** Complete — evidence recorded in v0.4.11

**Owner/worktree:** Codex integration owner, `feature/v0.4.0-d2-grpo-copilot`
**Type:** New training and verifier-authoring capability

## Problem and Scope

D1 proved the disposable-RunPod control plane with a GPU-free trainer. D2 must
prove the real path: a Qwen2.5-0.5B GRPO smoke run over a reviewed V1 NL-to-SQL
fixture, append-only local metrics, checkpoints published through `Storage` at
steps 50 and 100, and a kill/resume handoff through the same storage path.

This release also adds a laptop-only Verifier Copilot backed by OpenRouter's
explicit `x-ai/grok-4.5` model and a Docker-isolated validator for generated
verifier candidates. The Copilot produces reviewable proposals; humans approve
data and code before any GPU run.

## Non-goals

- No production-quality benchmark, gain claim, S3 storage, UI implementation,
  automated early stopping, or spurious-control experiment.
- No OpenRouter key, model weights, checkpoints, generated run data, or Docker
  image artifacts in Git.
- No direct OpenAI endpoint or host-process execution of untrusted Copilot code.

## Affected Areas and Contract Impact

- `trainer/`, `scripts/vf`, `core/storage/`, `core/rewards/`, and V1 fixtures.
- `app/api/`, `app/gpt/`, and a new `app/sandbox/` package.
- `docs/model-trainer/`, `docs/infrastructure/`, `docs/verifiers/`,
  `docs/gpt-integrations/`, `docs/backend-api/`, and `docs/evaluation-serving/`.
- Existing `core.contracts` remains additive and unchanged. Copilot request and
  response models live in the API module; `Metrics` keeps its existing shape.

## Validation and Merge Gate

1. `pytest -q`, `bash -n scripts/vf`, and isolated sandbox/API tests pass.
2. The 50 reviewed cases split deterministically into 40 train and 10 validation
   cases, and verifier safety tests reject unsafe SQL execution.
3. On RunPod, the GRPO launcher uses the provisioned
   torch 2.8.0+cu128 / vLLM 0.10.2 / verl 0.8.0 stack; checkpoints at 50 and 100
   are copied through `LocalStorage` and metrics are append-only.
4. After `vf kill`, a repeat `vf train <job> grpo_v1_0p5b` resumes from the
   newest Storage checkpoint and appends at the next step.
5. A curve image and final manifest sync without weights; no generated run,
   model, cache, or secret is staged.

## Completion Evidence — 2026-07-15

All five gates completed. The full `d2demo` GRPO smoke ran to step 50, was
stopped through `vf kill`, resumed only from the independently published
Storage checkpoint, and completed through step 100. `vf watch` produced a
hash-identical local metrics file, and the local API served its 100-point
curve. See `v0.4.11-d2-acceptance-evidence.md` in each affected doc area for
the exact command outcomes and interpretation limits.
