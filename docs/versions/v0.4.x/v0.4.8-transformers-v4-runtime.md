# v0.4.8 — Transformers v4 Runtime Compatibility

**Status:** In progress

**Owner/worktree:** Codex integration owner, `fix/v0.4.8-transformers-v4-runtime`

**Type:** D2 trainer runtime dependency bug fix

## Problem and Scope

The detached D2 two-step preflight passed dataset filtering, LoRA/FSDP model
construction, and vLLM launch preparation, but its vLLM tokenizer process
failed before the first rollout:

```text
AttributeError: Qwen2Tokenizer has no attribute all_special_tokens_extended
```

The persistent pod resolved `transformers==5.13.1`, while the pinned
`vllm==0.10.2` metadata only declares a lower bound (`transformers>=4.55.2`).
Pin a verified Transformers v4 release, install it in the persistent pod venv,
and repeat the detached two-step GRPO preflight.

## Non-goals

Do not change the model, CUDA/PyTorch/vLLM/verl pins, reward logic, GRPO
hyperparameters, or relax the tokenizer error. This patch only prevents the
unbounded dependency resolver from selecting the incompatible Transformers 5
major release.

## Affected Areas and Validation

- `requirements-trainer.txt`
- `docs/model-trainer/v0.4.8-transformers-v4-runtime.md`
- `docs/infrastructure/v0.4.8-transformers-v4-runtime.md`

Validation must show the exact installed package versions, construct the Qwen
tokenizer with `all_special_tokens_extended`, and run the existing detached
two-step preflight through vLLM startup and the first reward computation.

**Contract impact:** None; `core/contracts.py` and API payloads are unchanged.
