# v0.4.9 — SDPA Padding Compatibility

**Status:** In progress

**Owner/worktree:** Codex integration owner, `fix/v0.4.9-sdpa-padding-compat`

**Type:** D2 trainer runtime bug fix

## Problem and Scope

The v0.4.8 two-step preflight successfully started vLLM, ran initial
validation, and wrote a real metrics record. Its first GRPO training step then
failed inside `verl.trainer.ppo.ray_trainer._compute_old_log_prob`:

```text
ModuleNotFoundError: No module named 'flash_attn'
```

The model uses PyTorch SDPA deliberately, but this verl release still imports
the FlashAttention padding helpers when converting the rollout batch to its
padding-free representation. Supply a tested PyTorch-compatible resolution for
that helper path and repeat the detached two-step preflight through an actual
training step and checkpoint publication.

## Non-goals

Do not install or claim FlashAttention CUDA kernels merely to satisfy a helper
import; do not change the model, CUDA stack, GRPO objective, reward logic, or
acceptance-step counts. Any fallback must preserve the helper input/output
contract and stay limited to the trainer runtime.

## Affected Areas and Validation

- trainer runtime bootstrap/entrypoint as needed for the compatibility path
- `docs/model-trainer/v0.4.9-sdpa-padding-compat.md`

Validation must show: initial validation metrics, at least one real GRPO step,
a non-empty trainer metrics stream, the native `global_step_2` checkpoint
marker, and the Storage-published `ckpt/step_2` layout.

**Contract impact:** None; no shared Pydantic or API fields change.
