# v0.16.0 — S3 storage resilience proof

**Status:** in progress
**Owner / worktree:** Codex / repository-root `main`

## Problem and scope

Local run storage proves the contract but cannot survive loss of a disposable
GPU pod. This version adds an S3 implementation of the existing `Storage`
ABC, selected only when `VF_STORAGE_BACKEND=s3`; local remains the default.
Durable checkpoints, metrics, and artifacts use S3 while a pod-local cache is
only a disposable materialization/staging area required by the unmodified
trainer.

The implementation covers atomic checkpoint publication, append-only metric
records, idempotent same-step replacement, artifact round trips, a moto test
suite, one real bucket round trip, and (only if permissions allow) a 100-step
0.5B kill/resume proof on the already-live training pod.

## Explicit non-goals

No `trainer/` source/configuration change, frozen-data change, model training
objective change, paid LLM request, bucket-policy mutation, or secret file is
in scope. `VF_STORAGE_BACKEND` defaults to `local`; existing jobs do not move.
AWS credentials are never committed, printed, stored on the pod, or copied
into evidence. An `AccessDenied`, 403, or scope failure is attempted once,
recorded verbatim as `OWNER-ACTION`, and skips the GPU proof.

## Design and validation

`S3Storage` owns an immutable object-generation prefix per checkpoint and
publishes an atomic manifest object last. Readers consider only committed
manifests, so interrupted uploads are invisible; an identical step can safely
replace its manifest. Metric appends create immutable ordered record objects,
not a read-modify-write JSONL blob. A short-lived local cache materializes
published data for existing LocalStorage-oriented callers but is not the
durable source of truth.

Validation is ordered: moto covers every ABC method, interrupted checkpoint
invisibility, append ordering, and idempotency; a real S3 bucket verifies
checkpoint download SHA, 50 metrics, and interrupted publication; then the GPU
proof kills around step 60 and resumes to 100 from S3. Evidence includes object
inventory and SHA values plus a before/after curve. A bucket permission wall
records an exact re-run command and does not retry.

**S7a result:** the backend publishes checkpoint/artifact manifests only after
all immutable generation objects arrive, reconstructs and verifies materialized
cache files by SHA-256, and preserves metric records as immutable
`metrics.jsonl/` objects. Moto exercises every Storage ABC method, a failed
second file upload, 50 append-only metric records, a fresh-cache checkpoint
SHA reconstruction, and the opt-in selector. Full regression validation is
`230 passed, 1 skipped`.

**S7b result:** the one permitted real-bucket round trip passed: a checkpoint
was restored byte-for-byte with SHA-256
`a60b4cf7a2129129e6cf8a181b435ebf04f1be49037a1e1c76d936bf958a64e9`,
50 metrics were recovered in order, 53 proof objects were observed, and an
intentionally interrupted second-file upload had no visible manifest. The
ignored evidence JSON has SHA-256
`060be6c5df1d3cfe30dcca70bd0f32f10eb64db3a1bdc8bbf8e1849ce66e73f0`.
`scripts.s3_roundtrip` now makes this exact bounded proof repeatable and emits
an `OWNER-ACTION` record rather than retrying a permission failure.
