# v0.16.1 — native checkpoint recovery for S3 proof

**Status:** native recovery passed; timed node-loss proof incomplete, 2026-07-17.
**Owner/worktree:** Codex on `main` in `/Users/tangyiq/dev/verifierforge`.

## Problem and scope

The S7 GPU proof reached step 50 but the pre-publication serving gate rejected
the 0.5B native checkpoint because its configured export contains tokenizer
files and `model_world_size_1_rank_0.pt`, not a Hugging Face safetensors file.
The failure is not an S3 permission failure: 49 immutable metric objects are
already in the bucket and no checkpoint manifest was published.

The final delivery instruction forbids changes under `trainer/`. This version
adds a narrow, tested recovery utility outside that directory. It accepts an
already-complete native `global_step_<n>` directory, creates the usual wrapper
shape, publishes it through `S3Storage.save_checkpoint`, verifies the
materialized resumable payload, and records hashes. It will then allow the
unchanged `vf train` resume path to prove S3 node-loss recovery.

## Non-goals

- Do not change trainer code, configs, frozen datasets, model weights, or the
  service publication gate.
- Do not claim that this 0.5B native checkpoint is vLLM-servable.
- Do not print, persist, or commit AWS credentials; recovery credentials remain
  session-scoped tmux environment values and are cleared on exit.

## Affected areas

- `scripts/`: native-checkpoint recovery command and safe tmux credential
  handoff support.
- `core/storage/`: consumed through its existing public Storage API only.
- `tests/`: moto-backed wrapper, manifest, integrity, and invalid-native tests.
- `docs/infrastructure/` and `docs/evaluation-serving/`: operational record.

No public contract or mock field changes are allowed.

## Validation plan

1. Unit-test recovery against moto with a native directory and verify the
   restored `step_<n>/global_step_<n>` shape and file identity.
2. On the GPU pod, inject credentials only into a detached recovery tmux
   process, publish the quarantined step-50 native checkpoint, and verify its
   S3 manifest / materialized hashes.
3. Restart the unchanged S3 `vf train` job, inject the authorized kill after
   step ~60, and resume to 100 through S3.

## Local result

`scripts.s3_native_recovery` wraps the native checkpoint through a temporary
symlink directory so the durable result remains
`step_<n>/global_step_<n>`. It validates model/optimizer/data state before and
after materialization, records deterministic file-level identities, and uploads
the prior failure log plus recovery evidence as S3 artifacts. The credential
helper starts this one operation in a dedicated tmux session without embedding
values in shell text. Focused tests: `19 passed`; full suite: `242 passed, 1
skipped`.

## Pod result

The helper published the quarantined `global_step_50`, re-materialized it from
S3, and matched a 15-file tree SHA-256 of
`e4c37b47f6b32e9e5444fa86385687032d316d43b8a51b6ee00b8b085c5ba737`. It
also preserved the first-attempt log as an S3 artifact. The unchanged trainer
then resumed from that path and reached native step 100, which proves this
native S3 resume path works. It does not complete S7c: the strict serving gate
again rejected a non-safetensors export and the intentional step-60 kill was
not performed before the timebox expired.
