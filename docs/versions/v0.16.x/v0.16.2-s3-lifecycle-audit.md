# v0.16.2 — S3 credential lifecycle audit repair

**Status:** planned before implementation, 2026-07-17.  
**Owner/worktree:** Codex on `main` in `/Users/tangyiq/dev/verifierforge`.

## Problem and scope

An S3-backed job was cleaned by `vf kill` with GPU memory verified at 0 MiB,
but its lifecycle file still read `{"storage_credentials":"injected"}`.
The session environment no longer exists, so no secret persists; however, the
audit marker fails to prove that fact.

This patch changes only the remote `vf kill` success path. After the recorded
process group, tmux session, and Ray have been stopped and GPU cleanup has
passed, it writes a value-free lifecycle record with
`storage_credentials=cleared` and `cleared_by=vf_kill` when an S3 lifecycle
file exists. It also suppresses the harmless no-tmux-server diagnostic emitted
by a preflight session check.

## Non-goals

No trainer, S3 backend, frozen data, credential payload, pod configuration, or
model behavior changes. A failed GPU cleanup must not mark credentials cleared.

## Affected areas and validation

- `scripts/vf`: post-cleanup lifecycle audit record.
- `scripts/s3_job_env.py`: quiet missing-session preflight.
- `tests/test_vf_entrypoint.py` and `tests/test_s3_job_env.py`: assert the
  value-free marker and stderr suppression.
- `docs/infrastructure/` and `docs/evaluation-serving/`: record semantics.

Run focused control-plane tests plus `pytest -q`; verify the current already
cleaned pod only after `nvidia-smi` confirms 0 MiB.
