# v0.9.0 — deterministic NL2SQL difficulty reprojection

**Status:** planned before implementation
**Owner / worktree:** Codex / repository-root `main`
**Predecessor:** `v0.8.0` fenced-SQL extraction, A3 upper-bound rejection

## Routing decision

Branch A fixed the observed format disease and moved the subset from too hard
to too easy: its completed v2 Gate A has pass@1 `0.64`, above the unchanged
`0.60` maximum. The approved runbook's Branch B selects per-seed examples with
pass counts closest to four and in `[1, 7]`, so it is the deterministic
difficulty correction for either side of the fixed Gate A band. Codex records
this as an implementation judgment: “still does not pass” in A3 includes an
upper-bound rejection as well as the explicitly illustrated `< 0.20` case.
This does not relax a threshold or make an external request.

## Problem and scope

v0.9.0 implements B1–B3. It builds an immutable 326-record probe population
from the 276 accepted candidates plus the 50 reviewed original seeds at Git
ref `78912f1`; runs each at `k=8` against pod-local vLLM; atomically preserves
per-prompt full-pass counts; and deterministically projects a new 50-row
candidate subset. Per seed, it selects the mixed (`1..7`) row nearest to four;
unmixed seeds are discarded and filled from a second-best mixed row of another
seed, with a maximum of two selected rows from any source seed. The new subset
is verifier-v2 rechecked before its B3 Gate A run.

## Explicit non-goals

- No OpenRouter call, `.env` transfer, model change, threshold change,
  trainer logic/configuration edit, freeze, tag, or GRPO launch.
- Do not overwrite `trainer/data/nl2sql_v1.jsonl` before a passing B3 Gate A
  and O5 freeze.
- Do not invent candidate rows, retry an invalid probe, or widen the mixed
  range. The only source data are the 276 candidates and 50 Git-bound seeds.

## Affected areas

- `scripts/`: offline population construction and deterministic reprojection.
- `core/rewards/nl2sql.py`: used as verifier v2 without another score change.
- `tests/`: population provenance, projection ordering/backfill/cap, verifier
  recheck, and atomic publication coverage.
- `docs/evaluation-serving/`, `docs/verifiers/`, `docs/infrastructure/`,
  `docs/model-trainer/`, and `docs/p0-run-sheet.md`.

## Acceptance and stop conditions

B1 preserves 326 per-prompt counts in `[0,8]`. B2 must use the fixed selection
rule, record all discarded/backfilled sources, and stop condition ③ applies if
discarded seed count exceeds 20 (or no compliant backfill exists). B3 runs only
on the projected 50 rows and uses the unchanged Gate A thresholds. B3 passing
permits O5; B3 failing is stop condition ③. Infrastructure exit `2` twice in
succession is stop condition ①.
