# v0.32.0 — Capacity-aware RunPod selection

**Status:** reserved; implementation pending — 2026-07-19
**Owner/worktree:** Codex on `main`
**Release type:** minor provisioner capability

## Problem and scope

RunPod creation currently submits all mapped GPU IDs in one request and only
reacts after a broad capacity failure. P-2.5 must query current provider stock
and price first, filter the owner-approved candidate set, rank live offers by
on-demand hourly price, and attempt one concrete model at a time. A bounded
candidate race must end in `no_capacity` and a durable FAILED audit, never a
polling/retry storm. The selected GPU ID, cloud type and hourly price must be
stored in the provision audit for report assembly.

The approved `small_ada` candidate set is RTX 2000 Ada, RTX 4000 Ada, L4 and
A40. Live price—not hardcoded order—determines each attempt. Blackwell/B-series,
RTX 50-series and `sm_120` remain blocked.

## Official interface evidence

The current [RunPod REST index](https://docs.runpod.io/api-reference/overview)
has Pod CRUD but no standalone GPU availability resource. The official
[GraphQL spec](https://graphql-spec.runpod.io/) documents authenticated
`gpuTypes(input: GpuTypeFilter)` at `POST https://api.runpod.io/graphql`.
`GpuType.lowestPrice(input: GpuLowestPriceInput)` exposes `gpuTypeId`,
`uninterruptablePrice`, `stockStatus`, `availableGpuCounts`,
`maxUnreservedGpuCount`, and `countryCode`; the parent exposes ID/display name,
memory, cloud support/prices and maximum GPU counts. Pod creation remains
`POST https://rest.runpod.io/v1/pods`.

## Non-goals

No training, fixed region, Blackwell enablement, spot bidding, serverless GPU
selection, new provider, UI redesign, credential logging or unbounded retry.
The single live proof may spend at most `$1` and terminates immediately.

## Affected areas and contracts

- `core/provisioning_contracts.py`: candidate mapping and additive selection
  metadata only; no existing field removal.
- `app/provisioning/runpod.py`: GraphQL inventory query, deterministic parser,
  price ranking, bounded sequential create fallback and `no_capacity`.
- `app/provisioning/orchestrator.py`: whitelisted selection metadata in the
  existing append-only provision audit.
- `tests/`: out-of-stock, third-candidate, all-empty, create-race fallback and
  clean FAILED convergence.
- Provisioner/infrastructure/design/run-sheet documentation plus README.

## Validation and stop conditions

First pass dry-run mocks and the full suite, then push code before any paid
mutation. Live proof: query → choose cheapest live offer → create one Pod →
DELETE immediately → GET absence plus raw `vf-auto-* = 0`. Stop without a
second paid create on auth failure, malformed inventory, spend estimate above
`$1`, failed deletion, or ownership mismatch. DoD permits tag
`provisioner-capacity-aware` only after evidence is committed.
