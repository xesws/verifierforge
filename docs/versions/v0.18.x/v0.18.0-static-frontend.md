# v0.18.0 — independent static product frontend

**Status:** implemented and browser-validated
**Owner / worktree:** Codex / repository-root `main`
**Predecessor:** `v0.17.2` reviewer sandbox and handoff

## Problem and scope

VerifierForge needs a polished, self-contained product surface for a three-minute
hackathon recording. v0.18.0 adds an independent Vite, React, and strict
TypeScript frontend under `frontend/`. It turns the committed D4 training and
held-out artifacts into a five-page product narrative without connecting to the
existing API or mock server. Browser-only interactions, including forge job
creation and shipping controls, persist locally.

The visual system is a bright, restrained floating-glass workspace with clear
evidence hierarchy, graphite typography, evidence green, primary blue, and
amber reserved for discovered or pending states.

## Explicit non-goals

- No changes to `core/`, `trainer/`, `app/`, `mock/`, `data/`, backend
  contracts, artifact files, or operational runbooks.
- No FastAPI, mock-server, external API, runtime network request, SSR,
  database, authentication, or real deployment endpoint.
- No invented training points, curve smoothing, conflation of the 10-row
  training monitor with the 60-row held-out evaluation, or guardian metrics.
- No claim that projected savings or demo arena comparisons are committed
  held-out evidence.

## Affected areas

- `frontend/`: static application, local fixtures, generated evidence data,
  reusable components, styles, and validation configuration.
- `docs/frontend/v0.18.0-static-product-demo.md`: frontend-specific product,
  evidence, responsive, and visual acceptance record.
- `docs/versions/v0.18.x/`: version reservation and delivery scope.

## Contract impact

None. The frontend consumes repository artifacts at build-time only and does
not import, rename, or extend backend contracts. Generated JSON becomes a
frontend-local typed data source.

## Static data strategy

`frontend/scripts/generate-demo-data.mjs` reads the 400-point main metrics
JSONL, 200-point random-reward control JSONL, held-out report, and artifact
manifest. It retains every source `step`, `reward_mean`, `pass_at_1`, `entropy`,
and `timestamp`, then writes a committed
`frontend/src/data/generated/trainingMetrics.json`. The build regenerates this
file, while verified held-out values stay separate from product-only scenario
fixtures in `productScenario.ts`.

## Five-page user flow

1. **Discover** presents all three cost clusters, with Data Pull SQL as the
   proven live opportunity and direct paths to proof or forging.
2. **Forge** pre-fills a cluster from the URL, creates a queued browser-local
   job, confirms that no backend request occurred, and links to its local view.
3. **Run detail** shows the full main training monitor against the control that
   naturally ends at step 200, with quality, reward, and entropy views.
4. **Proof report** separates held-out checkpoints from training monitoring,
   explains step 350 selection, and labels savings and arena samples as product
   projections.
5. **Ship and Guard** simulates routing locally and leaves guardian telemetry in
   an honest awaiting-traffic state.

The default route redirects to `/discover`.

## Responsive requirements

The desktop workspace targets approximately 1440 px with a floating left rail,
wide evidence charts, and deliberate whitespace. At 1024 px, layouts compact
without hiding evidence. At 390 px, navigation becomes a bottom floating bar,
cards stack, chart controls and axes remain readable, touch targets remain
usable, and no horizontal overflow is permitted.

## Visual acceptance criteria

- Fog-gray, ice-blue, and faint mint radial background with subtle noise.
- Translucent white glass, 24–32 px blur, soft low-contrast shadows, and a
  meaningful hierarchy rather than a uniform card grid.
- Graphite `#17212B`, primary blue `#087CF0`, evidence green `#00A67E`, with
  accessible English UI copy and IBM Plex Mono data labels.
- Entrance, chart-draw, count-up, and restrained hover motion with full
  `prefers-reduced-motion` support.
- Keyboard focus visibility, WCAG AA text contrast, accessible chart legends,
  axes, tooltips, and controls.

## Validation plan

Run dependency installation, deterministic data generation, lint, and the
production build from `frontend/`. Verify that generated row counts are exactly
400 and 200 and that the control has no point beyond 200. Exercise the five
routes, persisted local state, responsive layouts, reduced motion, keyboard
focus, chart tabs and tooltips, arena expansion, routing controls, and browser
console. Compare each required public evidence value to the committed held-out
report and manifest before completion.

## Delivery result

The standalone frontend now implements all five routes and the requested shared
component set. The generator reproduced 400 main rows and 200 control rows,
with final steps 400 and 200 respectively, and retained the held-out selection
at step 350. `npm run generate:data`, `npm run lint`, and `npm run build` pass.

Browser validation covered 1440×900, 1024×768, and 390×844. All tested views
have document width equal to viewport width. Discover keeps all three clusters
in one desktop frame; the mobile rail moves to the bottom; the mobile training
chart remains readable at 336 px; Forge-created jobs and Ship routing state
survive refresh; Arena expands from two to seven samples; and the checked
browser console has no warning or error. Required screenshots are committed
under `frontend/screenshots/`.
