# v0.15.2 — serving runtime recovery and endpoint handoff

**Status:** in progress  
**Owner / worktree:** Codex / repository-root `main`

## Problem and scope

The v0.15.0 L4 serving pod received the canonical selected step-350 export and
passed the tree-identity gate, but vLLM 0.10.2 stopped during tokenizer loading:
`AttributeError: Qwen2Tokenizer has no attribute all_special_tokens_extended`.
This patch aligns only `transformers` and `tokenizers` on the serving pod to
the known-good training-pod values, while leaving vLLM and torch unchanged.

S1 archives exact package inventories from laptop, training pod, and serving
pod. S2--S4 perform at most three minimal dependency-alignment attempts and
preserve evidence. On a healthy local service, S5 validates the public RunPod
route with the official OpenAI SDK, and S6 temporarily sends synthetic
data-pull traffic through the real tuned endpoint before restoring canary to
zero.

## Explicit non-goals

This does not change `trainer/`, frozen data, checkpoint bytes, verifier
logic, contracts, model weights, or pricing. It makes no paid LLM request.
The endpoint key remains in ignored local `.env`, is rotated after any log
exposure, and is never written to Git, evidence, a command transcript, or the
pod filesystem.

## Affected areas and validation

Affected areas are serving infrastructure, evaluation/guardian evidence,
backend proxy routing, and the public-SDK proof asset. Contract impact is
**none**. The acceptance ladder is: matching dependency evidence; local
`/v1/models`; one local completion; public completion; 50/50 route evidence;
guardian points from real model responses; then canary zero. A compatibility
failure after three alignment attempts records the original error and skips
S5--S6 without blocking later waves.

Each attempt records package versions, the sanitized vLLM log, service state,
and outcome in the P0 run sheet. The serving process stays warning-level and
no command that could echo the API key is captured.
