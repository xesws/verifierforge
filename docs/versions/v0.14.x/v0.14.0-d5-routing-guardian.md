# v0.14.0 — D5 routing and guardian

**Status:** complete
**Owner / worktree:** Codex / repository-root `main`

## Problem and scope

v0.13.0 produced local, clusterable proxy traffic but could not direct a
cluster to a tuned endpoint or observe live quality. This version adds a
SQLite-backed routing state, an API compatible with the existing
`RoutingState` and `LivePassRate` contracts, and a best-effort asynchronous
guardian for routed SQL traffic.

The storage table uses `cluster_id`, `enabled`, `canary_percent`, and
`target_upstream`. The API deliberately maps `target_upstream` to the existing
contract's `target_model`; no shared contract field changes. Proxy requests
hash their system prompt to find a configured cluster. When a route is enabled
and its deterministic canary draw is below the percentage, the request uses
the selected tuned upstream; otherwise it uses the configured default.

`VF_PROXY_TUNED_UPSTREAM=fake-tuned` is the only tuned target exercised here.
It returns a response distinguishable from `fake`; no real checkpoint endpoint
is deployed or contacted. Every traffic row records `route_path` as `default`
or `tuned`.

The guardian runs out of the proxy response path. At configurable sampling
(default 20%), a tuned `data-pull-sql` completion is scored with the existing
`NL2SQLVerifier`; successful scores become rolling `LivePassRate` points.
Failures to schedule, load data, or score never delay or change a completion.

## Explicit non-goals

No trainer, model endpoint deployment, real upstream call, frozen-data change,
contract-field change, mock fixture redesign, or D4 rerun is in scope. Support
and invoice traffic may route but receive no SQL verifier score. This version
does not add authentication, migrations, or a database service.

## Affected areas and validation

Affected code: `app/proxy/`, `app/api/main.py`, `mock/server.py`, and focused
tests. Affected documentation: this version record plus backend-API and
evaluation-serving records. Contract impact is **none**; mock and real API must
validate against identical `RoutingState`/`LivePassRate` shapes.

Acceptance uses fake modes only: PUT the `data-pull-sql` route to 50% canary,
send 200 traffic-generator requests, verify both route paths are approximately
50/50 and are persisted, verify live guardian points, then set canary to 0 and
verify no tuned traffic. Run `pytest -q`, including mock/real shape tests, then
commit and push. Stop after this version; endpoint deployment and real upstream
integration remain for the D5 owner tomorrow.

## Result

The v0.14.0 acceptance used only `fake` and `fake-tuned` on localhost. After
the real API wrote a 50% `data-pull-sql` route, a 200-request replay completed
200/200 and persisted 113 tuned versus 87 default requests (56.5% tuned).
The guardian recorded 113 contract-shaped points with a latest rolling exact
pass rate of 1.0. Setting canary to zero, then replaying 20 more requests,
left the tuned count at 113: all 20 new requests used the default path.

No deployed model endpoint or real provider was contacted. The final suite was
**225 passed, 1 skipped**, including real/mock contract-shape coverage.
