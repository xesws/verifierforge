# v0.11.0 — D4 multi-GPU main training

**Status:** M0 documentation gate; M1–M6 not started
**Owner / worktree:** Codex / repository-root `main`
**Starting point:** `4306aeb` after the immutable
`v0.10.2-p0-three-piece-freeze` tag and completed 0.5B Gate B smoke

## Problem and scope

The frozen 50-row training pool and disjoint 60-row held-out set are admitted,
but the only real train so far is a single-L4, 0.5B, 100-step Gate B smoke.
D4 promotes that path to an explicit 1.5B multi-A100 main experiment, with a
spurious random-reward control and a held-out-only before/after result.

M1 moves the disposable compute worker while retaining the `/workspace`
network volume; M2 proves FSDP plus vLLM rollout/weight synchronization before
the main run; M3 launches a 400-step, `k=8` main job and saves every 50 steps;
M4 launches a 200-step 0.5B random-reward control concurrently; M5 evaluates
the frozen held-out set after training; and M6 keeps all public metrics and
artifacts in `Storage` and returns them through `vf watch`.

## Fixed experiment decisions

- The target must expose **at least three A100 GPUs** in the same RunPod region
  as the mounted network volume. The 1.5B main job reserves at least two GPUs;
  the 0.5B control reserves a different GPU. If that isolation cannot be
  established, M1 stops rather than silently serializing M4.
- The main configuration is `Qwen/Qwen2.5-1.5B-Instruct`, 400 steps, `k=8`,
  and a 50-step checkpoint cadence. The frozen training-pool alias remains
  byte-identical to its tagged identity; no data, verifier, or contract field
  changes are permitted.
- A checkpoint is retained only after a held-out comparison. The held-out set
  never enters gradients, rollout rewards, or trainer validation. Tie-breaking
  for equal held-out pass@1 is the lower checkpoint step.
- The entropy brake is an explicit, tested trainer policy: after step 20 it
  stops the job when ten consecutive bridged entropy values are each below
  25% of the median of the first ten values. Its stop artifact names the
  window, threshold, and last checkpoint; it does not fabricate a successful
  final model.
- The control uses a deterministic fixed-seed Bernoulli(0.5) reward adapter,
  not NL2SQL verifier reward. It has a separate job ID, model, metrics, and
  checkpoint namespace, so its curve is evidence against spurious learning,
  not a production score.
- The fixed held-out before snapshot remains exactly
  `pass@1=0.5833333333333334`, `pass@8=0.7666666666666667`, and
  `mixed_fraction=0.4666666666666667` from v0.10.1. Every gain or DoD claim
  compares only to that set.

## Affected code and documents

- `trainer/grpo_config.py`, `trainer/grpo_train.py`, `trainer/grpo_reward.py`,
  `trainer/launch.sh`, new operator-visible YAML configurations, and focused
  tests add GPU allocation, main/control configuration, entropy stopping,
  checkpoint-selection evidence, and random reward behavior.
- `scripts/vf` gains only the remote environment/allocation plumbing needed to
  launch isolated jobs and retains its preflight/detached-tmux discipline.
- `core/storage/` changes only if AWS credentials are available in the first
  30 minutes and an S3 implementation can be tested without delaying M3;
  otherwise LocalStorage is the recorded main-run backend and a Gate-B-scale
  S3 proof is deferred.
- The matching trainer, infrastructure, evaluation-serving, and verifier
  documents below are part of this version. `docs/p0-run-sheet.md` remains the
  recovery context and is updated after each completed M-step.

## Explicit non-goals and contract impact

- No dataset reprojection, new augmentation, verifier-tier change, Gate-A
  threshold change, model API call, secret transfer, or frontend/API contract
  change.
- No automatic RunPod lifecycle orchestrator is introduced. The existing
  `vf` control plane operates an already-running SSH endpoint; lifecycle
  actions require a usable RunPod control credential/tool and are recorded as
  operator actions.
- No S3 use is claimed without an actual read/write proof. Missing AWS
  credentials are not a reason to delay the LocalStorage main run.

## Validation and stop conditions

1. M1 verifies the old L4 is stopped only after the replacement multi-A100 pod
   has been provisioned with the same volume, `vf bootstrap` succeeds, and
   `nvidia-smi` shows the intended isolated GPUs. Missing lifecycle access,
   wrong region/volume, fewer than three A100s, or bootstrap failure stops
   before M2.
2. M2 runs 30 1.5B steps. It must log FSDP across the reserved GPU count,
   vLLM rollout startup and at least one weight-update/rollout cycle, append
   Storage metrics, and produce visible curve points. Any NCCL, OOM, worker
   synchronization, Storage, or throughput failure stops before M3/M4.
3. M3/M4 start only after M2 passes. The main job must make checkpoints at
   50-step intervals; the control must use a distinct GPU and half the main
   step count. A job failure or entropy brake preserves artifacts and stops
   its post-run claim; it is never silently retried with altered settings.
4. M5 writes a hash-bound held-out after-evaluation and report artifact next to
   the fixed before numbers. A trained checkpoint can be called best only
   after this measurement. Evaluation failure yields no gain claim.
5. M6 keeps `Storage` append-only and `vf watch` synchronization active. The
   AWS credential check records only presence/readiness, never values. If not
   ready by the 30-minute boundary, LocalStorage remains selected and the S3
   proof moves to a small Gate-B-scale job after the main result.
