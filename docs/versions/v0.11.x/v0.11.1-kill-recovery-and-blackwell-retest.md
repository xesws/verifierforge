# v0.11.1 — Kill recovery and Blackwell smoke retest

**Status:** planned; documentation gate complete before implementation  
**Owner / worktree:** Codex / repository-root `main`  
**Starting point:** `dab2d5d`, where M2′ stopped without a rollout or metric

## Problem and scope

M2′ left a Ray worker after `vf kill` ended tmux, and the 1.5B Blackwell smoke
did not reach a rollout/metric/checkpoint within its timebox. v0.11.1 has two
separate workflows. W1 unconditionally makes `vf kill` own the whole remote
job lifecycle: launch in a recorded `setsid` process group, terminate that
group with TERM then KILL, force-stop Ray on the dedicated pod, and fail if GPU
memory remains allocated. W2 first collects read-only evidence for fifteen
minutes, then classifies the block as resource waiting, engine initialization
hang, or communication initialization hang.

## Decisions, non-goals, and contract impact

- For W2 only, resource declarations may be made single-GPU consistent; an
  engine-init hang may enable `enforce_eager` and use documented attention
  backend fallbacks. No other trainer behavior, data, verifier, contract,
  secrets, model API, or frozen-file identity may change.
- W2 diagnosis, repair, and retests have a combined 45-minute timebox. At most
  two retests are allowed. Any other classification, failed permitted repair,
  or second hang stops for human direction.
- M3–M6 remain blocked until the retest proves rollout, a post-update path,
  Storage metrics, a curve point, throughput, and clean teardown.

## Affected areas and validation

- `scripts/vf`, a focused test, and the matching infrastructure document cover
  process-group launch/teardown and the real fake-job acceptance check.
- The Blackwell smoke configuration/launch path and matching model-trainer
  document record exact config dumps, diagnostic evidence, classification, and
  any permitted fallback.
- `docs/p0-run-sheet.md` is updated after W1, diagnosis, each retest, and the
  final stop/pass decision. Run `pytest -q`, shell syntax checks, and the
  remote fake-job W1 test before W2. Preserve logs/artifacts under ignored
  `runs/`; never print or copy `.env` values.

## W1 result

Commit `537e505` implemented the recorded `setsid` PGID boundary and guarded
GPU cleanup. The real remote `fake_smoke` job `w1-kill-v0111` recorded PGID
`9991` and produced 39 metrics records before `vf kill`. The command completed
with `GPU memory: 0 MiB`; the Ray fallback reported that no Ray process was
active. A separate remote assertion verified tmux absent, job process absent,
compute applications absent, and zero GPU memory. W1 therefore passes and W2
may begin as a read-only diagnostic only.

## W2 diagnosis and first permitted repair

The 15-minute read-only job `p0-w2-diag-v0111` did not produce a metric,
rollout, checkpoint, or throughput. The raw default `ray status` returned
`ConnectionError: Could not find any running Ray instance`; the same command
against the observed GCS address `172.23.0.2:42207` timed out with exit `124`.
Ray itself was live: its raylet advertised exactly one GPU and the active actor
dump resolved to `n_gpus_per_node=1`, TP `1`, GPU memory utilization `0.5`, and
`enforce_eager=False`. A `ray::WorkerDict` held 7,192–8,286 MiB with 0% GPU
utilization; the vLLM server command stopped after its `enable_sleep_mode` log
without readiness, rollout, or error.

This is classified as an engine-initialization hang. The first and only next
repair is to set the active actor rollout's `enforce_eager=True`, bypassing CUDA
graph capture. It is not a resource fix and does not change the frozen data,
model, verifier, reward, or TP/GPU declarations. One retest follows; a second
hang permits attention-backend fallback only under the existing two-retest/
45-minute stop rule.

The eager retest `p0-w2-eager-r1-v0111` proved the override was active, but
after roughly 14 minutes it reached only a 7,192–8,286 MiB WorkerDict and never
created vLLMHttpServer, a rollout, a metric, or a checkpoint. W1 stopped it
cleanly at 0 MiB. The installed vLLM source exposes `_Backend.TORCH_SDPA`; the
second and final retest therefore injects only
`VLLM_ATTENTION_BACKEND=TORCH_SDPA` into both driver and Ray runtime worker
environments. Eager mode remains enabled and all model/data/TP/GPU declarations
remain unchanged. A second hang or the 45-minute total W2 deadline stops.

## W2 final stop

The final job `p0-w2-sdpa-r2-v0111` proved both
`enforce_eager=true` and `VLLM_ATTENTION_BACKEND=TORCH_SDPA` were passed to the
launch/Ray runtime. Ray started at `19:57:10 UTC`, but before the 45-minute W2
deadline it never acquired GPU compute work (3 MiB, 0% utilization), created a
WorkerDict/vLLM server, or produced rollout, throughput, metrics, or a
checkpoint. It logged no CUDA/OOM/Traceback diagnostic. W1 terminated it at
`20:03:58 UTC` and verified zero GPU memory.

The eager and SDPA retests are the two allowed W2 attempts. v0.11.1 therefore
stops here: no third backend, config workaround, data change, M3, M4, M5, or M6
may start without human approval. The three ignored `runs/p0-w2-*` directories
are the retained evidence set.
