# v0.6.2 — pod-local Gate A recovery

**Status:** in progress
**Owner / worktree:** Codex / repository-root `main`
**Predecessor:** `1594544` (`v0.6.1 Eval: record Gate A provider stop`)

## Problem and scope

The first Gate A run used the augmentation client configuration and failed without
recoverable provider evidence. v0.6.2 separates evaluation configuration from
augmentation/copilot, makes completion failures auditable, and moves Gate A to
a local vLLM server on RunPod. It preserves the already verifier-screened full
dataset and deterministic 50-row subset.

## Explicit non-goals

- No changes to contracts, mock/API/frontend shapes, verifier scoring, seed
  data, trainer Python logic, GRPO configuration, or trainer data-path config.
- No OpenRouter key, `.env`, tunnel, or external completion request on the pod.
- No automatic whole-run Gate A retry, threshold relaxation, training before a
  passing subset, or claim of held-out model improvement.

## Affected areas

Matching v0.6.2 documents exist under GPT integrations, evaluation serving,
infrastructure, model trainer, and verifiers. The durable operational state is
`docs/p0-run-sheet.md`.

## Public interface and validation

`VF_EVAL_BASE_URL` and `VF_EVAL_MODEL` are mandatory for Gate A; optional
`VF_EVAL_API_KEY` may override a non-secret local placeholder. `--report`
becomes mandatory and `--reference` produces a full-set measurement without
threshold-based exit failure. Tests cover config isolation, evidence/error
chains, retry/circuit behavior, v2 manifest binding, and the full pytest suite.
