# v0.3.0 — RunPod Runtime Provisioning

**Status:** Planned — D1 gate passed; implementation may begin after this document is committed.

**Owner/worktree:** `main` integration worktree

**Type:** trainer-runtime provisioning feature

## Problem and Scope

Prepare the persistent RunPod volume for tomorrow's real-training smoke without running any training today. Start detached downloads of `Qwen/Qwen2.5-0.5B-Instruct` and `Qwen/Qwen2.5-1.5B-Instruct` into `/workspace/hf-cache`; make a best-effort detached download of `openai/gpt-oss-20b`; and add the requested `verl` and `vllm` runtime dependencies to `requirements-trainer.txt`. The resolved attempt pins `vllm==0.25.1` and the official verl `release/v0.8.0` commit `cb367c201187259e109a90f58af9ad9659126429`, using PyTorch's CUDA 12.8 index. Attempt dependency installation in a named tmux session, capped at 45 minutes from its start.

## Non-goals

No GRPO run, reward adapter, model serving endpoint, training configuration, checkpoint-format change, contract change, Docker image, or unbounded dependency debugging. A failed or timed-out install is an expected outcome and must be recorded rather than retried indefinitely.

## Affected Areas

- `requirements-trainer.txt`
- `scripts/vf` and `trainer/bootstrap.sh` for deploy-key restoration only
- `NOTES.md` only if the dependency attempt fails or times out
- `docs/infrastructure/v0.3.0-runpod-runtime-provisioning.md`
- `docs/model-trainer/v0.3.0-runpod-runtime-provisioning.md`

## Contract Impact

None. The shared job, storage, API, and mock contracts remain unchanged.

## Execution Plan

1. Record the remote Python, CUDA driver/runtime, Torch, and pip resolution context. The preflight found L4 / driver 570.195.03 / CUDA 12.8 / Python 3.12.3 and no Torch in the project venv.
2. Commit and push the trainer requirement update before the pod pulls it. Install `vllm==0.25.1` first through the CUDA 12.8 PyTorch index, then install the pinned official verl source with `--no-deps` so it cannot silently replace the resolved inference stack.
3. Persist the already-authorized read-only deploy identity under `/workspace/.ssh/`. The RunPod network volume reports private-key files as mode 0666, which OpenSSH refuses directly; before each Git pull/clone, `vf bootstrap` and `bootstrap.sh` must copy it into an ephemeral mode-0600 `/root/.ssh/` file. Do not record or commit key material.
4. Start separate, detached tmux sessions for model-cache downloads and the timeboxed dependency install. Bootstrap `huggingface_hub` only inside the download session if it is not already available. Every log, pip cache, virtual environment, and downloaded cache path must live under `/workspace`.
4. Poll the sessions without attaching permanently. Preserve successful downloads; on failure, keep logs and stop only the failed operation.
5. Verify import versions only if installation succeeds. Record exact outcomes and any failure in the area documents and `NOTES.md`.

## Validation and Merge Gate

`requirements-trainer.txt` must parse under pip; the pod must pull the committed revision; each tmux session must have a persistent log; cache paths must resolve beneath `/workspace/hf-cache`; and no secrets, model weights, or virtual-environment artifacts may enter Git. The dependency command must be terminated at 45 minutes or earlier if it reaches an unrecoverable CUDA/version conflict.
