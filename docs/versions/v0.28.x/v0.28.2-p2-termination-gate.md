# v0.28.2 — P-2 termination gate correction

**Status:** retry implementation complete; live retry pending — 2026-07-19
**Owner/worktree:** Codex on `main`
**Release type:** patch correction to the v0.28.0 RunPod P-2 capability

## Problem and scope

The v0.28.0 executor incorrectly made a delayed RunPod billing row part of the
synchronous teardown gate. The provider accepted DELETE for the gold pod and a
fresh account inventory now contains zero `vf-auto-*` names, yet the executor
stopped after 15 minutes because billing history remained empty.

This patch makes deletion acceptance plus a raw `GET /pods` prefix-zero
readback the blocking termination proof. Billing becomes a durable, idempotent
reconciliation at deletion `+1h` and `+6h`; it is appended when available but
does not block the orphan proof, the 100-step P-2 run, or the semantic tag.

The existing gold evidence is resumed rather than creating a second paid gold
pod. The orphan probe and approval-driven 0.5B run remain mandatory.

The first two orphan creates after this correction returned HTTP 500 with
`There are no instances currently available` for all four approved Community
GPU IDs; neither attempt allocated a pod. A read-only stock query showed no
Community price/stock for those IDs, while Secure reported low stock for the
same RTX 4000 Ada (`$0.26/hr`) and L4 (`$0.39/hr`) IDs. The adapter may therefore
retry once in Secure only after that exact Community capacity error. GPU order,
on-demand mode, `$5` wave fuse and 180-minute runtime fuse remain unchanged.

The live Secure pod exposed no `runtime` object and formatted `lastStartedAt`
as `2026-07-19 10:22:16.76 +0000 UTC`. Python's ISO parser rejected the trailing
zone label, so the first local lifecycle polls reported zero uptime/cost even
though the provider returned `$0.39/hr`. This patch must normalize that exact
provider timestamp shape; the already-running 100-step job remains protected
by an external wall-clock check at the same 180-minute/$5 limits.

The first full attempt reached step 49 and staged step 50, then failed closed
before S3 publication with `ServingSmokeError: checkpoint export conversion
failed: missing safetensors index`. The 0.5B exporter correctly produced one
`model.safetensors`; the older converter only accepted a sharded export with
`model.safetensors.index.json`. The trainer child was terminated, the pod was
deleted, the target reads absent and raw `vf-auto-*` count is zero.

The compatibility repair may accept exactly one safetensors file when no index
exists, hash that sole source file as its identity, and still generate a
standard output index before vLLM smoke. Multi-shard exports without an index
remain invalid. The retry resumes after the already-passed gold/orphan stages,
uses a clean `-r2` job/S3 prefix, and retains the approval's original terminal
handle binding plus an explicit retry audit; it must not overwrite that binding
or mix the 49 old metrics into the new run. Prior spend is conservatively
reserved from elapsed wall time at the maximum allowed wave burn rate.

## Non-goals

- Do not change trainer logic, frozen data, model configuration, public
  contracts, feature flags, or the product approval-to-execution boundary.
- Do not touch any pod whose name does not start with `vf-auto-`.
- Do not treat an absent billing row as zero cost.
- Do not repeat the historical gold create/status/delete probe.
- Do not use spot capacity, add GPU IDs, or fall back to Secure for an auth,
  validation, identity or transport error.
- Do not publish the failed step-50 staging tree, reuse its absent checkpoint,
  or silently rebind the approval to a replacement provider handle.

## Affected areas and contract impact

- `scripts/provision_runpod.py`: split deletion confirmation from billing,
  resume immutable gold evidence, preserve conservative budget accounting and
  expose an idempotent billing reconciliation command.
- Provisioner/infrastructure/database evidence: record deletion, raw prefix
  inventory, pending/observed billing, estimated cost and S3 identities.
- Model trainer: execution evidence only; no code or config change.
- Submission/current-state documents: updated only if the complete live DoD
  passes.

No Pydantic, API, mock or storage contract changes are required. The existing
provider-neutral `terminate() -> None` protocol remains unchanged.

## Validation plan and stop conditions

Unit tests prove that billing absence does not block a clean deletion,
that any raw `vf-auto-*` record does block, that gold resume creates no second
gold pod, that reconciliation is idempotent and that estimated cost still arms
the `$5` fuse. Retry tests additionally prove immutable provider binding,
required terminal/deletion audit evidence, conservative prior-spend accounting,
and an empty `-r2` S3 prefix. Clean `HEAD == origin/main` remains mandatory
before paid action.

Each orphan/training resource must pass DELETE, target-absent and raw
prefix-zero readback. Ownership mismatch, prefix residue, budget/runtime fuse,
S3 incompleteness or SHA mismatch stops the wave and prevents
`provisioner-p2-live`. Billing delay alone does not.

## Implementation validation

The termination module keeps the raw-prefix deletion proof and delayed billing
schedule separate from the live CLI. After adding the exact-error Secure
fallback, focused provisioning tests passed `15` tests; the full suite passed
`393 passed, 1 skipped` after the timestamp parser regression was added. No
paid resource was created during implementation or tests.

The fail-closed retry repair accepts a sole `model.safetensors`, hashes it as
the source identity and still emits a normal index before vLLM smoke; multiple
unindexed shards remain rejected. Focused retry/export/serving tests passed
`35` tests and the complete pre-live suite passed `398 passed, 1 skipped`.
