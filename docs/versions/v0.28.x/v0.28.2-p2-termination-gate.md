# v0.28.2 — P-2 termination gate correction

**Status:** implementation complete; live orphan/training proof pending — 2026-07-19
**Owner/worktree:** Codex on `main`
**Release type:** patch correction to the v0.28.0 RunPod P-2 capability

## Problem and scope

The v0.28.0 executor incorrectly made a delayed RunPod billing row part of the
synchronous teardown gate. The provider accepted DELETE for the gold pod and a
fresh account inventory now contains zero `vf-auto-*` names, yet the executor
stopped after 15 minutes because billing history remained empty.

This patch makes deletion acceptance plus a raw `GET /pods` prefix-zero
readback the blocking termination proof. Billing becomes a durable, idempotent
reconciliation at deletion `+1h` and `+6h`; it is appended when available but
does not block the orphan proof, the 100-step P-2 run, or the semantic tag.

The existing gold evidence is resumed rather than creating a second paid gold
pod. The orphan probe and approval-driven 0.5B run remain mandatory.

## Non-goals

- Do not change trainer logic, frozen data, model configuration, public
  contracts, feature flags, or the product approval-to-execution boundary.
- Do not touch any pod whose name does not start with `vf-auto-`.
- Do not treat an absent billing row as zero cost.
- Do not repeat the historical gold create/status/delete probe.

## Affected areas and contract impact

- `scripts/provision_runpod.py`: split deletion confirmation from billing,
  resume immutable gold evidence, preserve conservative budget accounting and
  expose an idempotent billing reconciliation command.
- Provisioner/infrastructure/database evidence: record deletion, raw prefix
  inventory, pending/observed billing, estimated cost and S3 identities.
- Model trainer: execution evidence only; no code or config change.
- Submission/current-state documents: updated only if the complete live DoD
  passes.

No Pydantic, API, mock or storage contract changes are required. The existing
provider-neutral `terminate() -> None` protocol remains unchanged.

## Validation plan and stop conditions

Unit tests must prove that billing absence does not block a clean deletion,
that any raw `vf-auto-*` record does block, that gold resume creates no second
gold pod, that reconciliation is idempotent and that estimated cost still arms
the `$5` fuse. Run focused tests, full `pytest -q`, secret scanning and clean
`HEAD == origin/main` before any paid action.

Each orphan/training resource must pass DELETE, target-absent and raw
prefix-zero readback. Ownership mismatch, prefix residue, budget/runtime fuse,
S3 incompleteness or SHA mismatch stops the wave and prevents
`provisioner-p2-live`. Billing delay alone does not.

## Implementation validation

The termination module keeps the raw-prefix deletion proof and delayed billing
schedule separate from the live CLI. Focused provisioning tests passed `13`
tests; the full suite passed `390 passed, 1 skipped`. No paid resource was
created during implementation or tests.
