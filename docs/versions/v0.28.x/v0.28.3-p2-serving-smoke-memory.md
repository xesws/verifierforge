# v0.28.3 — P-2 in-process serving-smoke memory

**Status:** implementation complete; final live attempt pending — 2026-07-19
**Owner/worktree:** Codex on `main`
**Release type:** patch correction to the P-2 checkpoint publication gate

## Problem and scope

The v0.28.2 retry fixed the 0.5B single-file export and reached step 50. The
conversion succeeded and vLLM loaded the converted 0.927 GiB model, but the
loopback serving gate still used `--gpu-memory-utilization 0.10` while the
paused trainer retained about 15.7 GiB on the 24 GiB L4. vLLM reported
`Available KV cache memory: -5.32 GiB` and then `ValueError: No available
memory for the cache blocks`. The trainer failed closed, the replacement pod
was deleted, and raw `vf-auto-*` inventory returned to zero.

This patch raises the in-process serving-smoke ceiling to a bounded 0.90 so
vLLM may use the GPU memory left by the paused trainer; eager mode, max model
length 64 and max sequence count 1 keep the smoke workload small. It also
retains a longer vLLM error tail in published evidence and makes the local P-2
controller recognize the S3 failure artifact instead of waiting for a runtime
fuse after remote tmux exits.

One final clean `-r3` full-stage attempt is allowed. It must preserve the
original approval binding, validate the complete retry chain, reserve both
earlier attempts conservatively against the same `$5` wave, and use an empty
S3 prefix. Gold and orphan stages must not repeat.

## Non-goals

- No change to GRPO math, frozen data, model, steps, k, checkpoint interval,
  provider GPU allowlist, spot policy, public contracts or feature flags.
- No relaxation of the vLLM load plus real-completion publication gate.
- No reuse of either failed attempt's metrics or staging checkpoint.
- No fourth attempt if the clean `-r3` run fails.

## Affected areas and contract impact

- `trainer/serving_smoke.py`: bounded memory ceiling and complete failure tail.
- `app/provisioning/live.py` and `scripts/provision_runpod.py`: remote failure
  observation and chained retry admission.
- Tests and P-2 evidence/run-sheet documents.

There is no API/Pydantic/storage contract change. The S3 failure artifact is
an existing key; the controller only starts observing it.

## Validation and stop conditions

Tests must cover the bounded memory setting, preserved root-cause text, S3
failure recognition, retry-chain identity, immutable approval binding,
conservative cumulative spend and empty `-r3` prefix. Full `pytest -q`, secret
scan and clean pushed HEAD are required before the paid attempt.

The live run stops on any provider residue, identity mismatch, non-empty S3
prefix, remote failure artifact, budget/runtime fuse, incomplete step-100
objects, serving smoke failure or SHA mismatch. Passing requires 100 metrics,
step-50 and step-100 checkpoint evidence as defined by the P-2 profile, final
model marker, curve PNG, verified collection, DELETE and raw prefix zero.
