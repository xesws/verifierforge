# v0.28.5 — P-2 post-training serving gate

**Status:** implementation validated; live attempt pending — 2026-07-19  
**Owner/worktree:** Codex on `main`  
**Release type:** patch correction to v0.28.4

## Problem and scope

The four P-2 training attempts reached the first checkpoint while verl/Ray still
owned most of the single GPU. The checkpoint bridge immediately launched a
second vLLM process, so the service gate failed on free-memory admission before
training could continue. A fixed `gpu_memory_utilization` cannot make two
same-GPU runtimes safely overlap.

For the P-2 profile only, completed training checkpoints become atomic
**candidates** in Storage during training. Candidate manifests retain file
names, sizes and SHA-256 identities but do not satisfy the published-checkpoint
or `COLLECTING` completion gates. After the trainer process exits successfully,
the launcher stops its Ray runtime and a separate finalizer selects step 100,
materializes it from Storage, performs conversion plus `/v1/models` and a real
completion on the now-exclusive GPU, and only then publishes the checkpoint
and final artifacts.

## Non-goals

- No GRPO algorithm, frozen data, reward, model, step count, `k`, checkpoint
  cadence, provider ordering, budget cap or approval contract change.
- No weakening of the X5 serviceability invariant.
- No mid-training pause/resume implementation in this patch.
- No sixth live attempt if the fifth attempt fails.

## Affected areas and contract impact

- `trainer/checkpoint_bridge.py`, `trainer/grpo_train.py`, a small post-training
  finalizer, P-2 config/launcher, and focused tests.
- `app/provisioning/live.py` and the P-2 S3 collector distinguish candidate
  checkpoint manifests from a service-accepted published checkpoint.
- Provisioner/run-sheet evidence and infrastructure documentation.
- Reviewer-facing README, JUDGES and submission documents after the live gate
  passes, so they no longer describe the historical v0.28.0 stop as current.

Public API and Pydantic contracts are unchanged.

## Validation and stop conditions

Tests must prove that training-time candidates never appear in published
checkpoint discovery, finalization invokes the gate only after training, a
failed gate cannot publish or complete collection, and S3 candidate identities
are collected. Full `pytest -q`, shell syntax, clean pushed HEAD, empty new S3
prefix, raw provider prefix zero and cumulative conservative spend below `$4.5`
are mandatory before provider create.

The fifth attempt must produce 100 metrics, candidate manifests, one
service-accepted step-100 manifest, final artifact, curve and SHA-verified
collection before deletion. Any gate, identity, budget, cleanup or completion
failure stops without the `provisioner-p2-live` tag.

## Implementation validation

The P-2 config alone selects `post_training`; the other six profiles retain
`per_checkpoint`. Candidate artifact and S3-manifest separation, finalizer
ordering, failed-gate publication refusal, candidate-aware collection and the
`$4.5` pre-create/runtime stop were covered by 53 focused tests. The complete
suite passed `411 passed, 1 skipped`; shell syntax, bytecode compilation,
tracked-secret scan and `git diff --check` also passed.

The collector additionally treats the service-evidence manifest as a required
completion object, independently verifies its file SHA, and rejects collection
unless it records models HTTP 200 plus a non-empty HTTP 200 completion. This
closes the gap between a published checkpoint and locally collected proof.
