# v0.10.2 — U3 three-piece data freeze

**Status:** immutable tag pushed; detached Gate B running
**Owner / worktree:** Codex / repository-root `main`
**Predecessor:** `2a432ac` recorded a passing U2 gate and held-out baseline

## Problem and scope

U2 now admits the deterministic 50-row training pool, and a disjoint 60-row
held-out baseline exists. Before any 0.5B Gate B run, the human requires one
immutable identity consisting of exactly the training pool, held-out set, and
verifier v2 provenance.

This version adds a small tracked freeze helper that verifies those identities,
their completed local-vLLM evidence, and zero source-record overlap; it writes
a canonical manifest. The freeze commit binds the runtime fixture as a
byte-identical alias of the frozen training pool, creates annotated tag
`v0.10.2-p0-three-piece-freeze`, and pushes it before Gate B starts.

## Explicit non-goals and contract impact

- Do not change contracts, verifier scoring, U2 thresholds, model/config,
  trainer logic, or GRPO configuration.
- Do not stage or alter the pre-existing untracked `scripts/freeze_nl2sql.py`;
  its old full/subset interface is outside this U3 scope.
- The fixture alias is not a fourth frozen dataset: its required SHA-256 equals
  the training-pool SHA-256, so Gate B consumes the exact frozen pool without a
  trainer data-path change.

## Affected code and documents

- `scripts/freeze_three_piece.py` and focused tests validate/publish the
  manifest and runtime fixture alias atomically per artifact.
- `data/nl2sql/` receives the canonical U3 manifest; `trainer/data/` receives
  the verified byte-identical training-pool alias.
- Infrastructure documents define tag/push, evaluator shutdown, detached Gate
  B launch, and two-minute status cadence.
- Evaluation documents bind both U2 evidence sets; verifier documents bind
  version/source/blob provenance; trainer documents retain the held-out-only
  before/after rule.

## Validation and stop conditions

Run focused freeze tests, `pytest -q`, exact reference-SQL verification,
manifest/fixture hash recomputation, then commit/tag/push. Any validation,
manifest, fixture, commit, tag, push, evaluator-release, or launch failure
stops the workflow. Gate B starts only from the tag commit and only against the
training-pool alias; all later before/after or DoD numbers use held-out only.
