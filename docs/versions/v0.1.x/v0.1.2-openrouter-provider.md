# v0.1.2 — OpenRouter GPT Provider Patch

**Status:** Implemented — live smoke test pending an operator-provided key

**Owner/worktree:** Codex, `feature/v0.1.2-openrouter-provider`

**Type:** provider and configuration patch

## Problem and Scope

The project currently has no GPT runtime client, and direct OpenAI billing is unavailable. Add a minimal `app/gpt/` client that sends OpenAI-compatible chat-completion requests through OpenRouter using `OPENROUTER_API_KEY` and `https://openrouter.ai/api/v1`. Keep model selection configurable through `VF_GPT_MODEL`; the default is the reviewed explicit slug `x-ai/grok-4.5`, while operators may deliberately override it for experiments.

Add the Python `openai` SDK only as a client library pointed at OpenRouter's base URL. Optional attribution headers come from non-secret environment variables. Do not load `.env` automatically or commit credentials.

## Non-goals

No direct OpenAI API calls, provider fallback, streaming, prompt generation, FastAPI route, verifier Copilot flow, task-expansion flow, report-narrative flow, or contract change. Those callers will integrate with the client under later version documents.

## Affected Areas

- `docs/gpt-integrations/v0.1.2-openrouter-provider.md`

## Validation and Merge Gate

Unit-test environment parsing, OpenRouter base URL, headers, model selection, and response extraction with an injected fake client; no live key is required. A manual smoke test may use `OPENROUTER_API_KEY` from the process environment, never a committed `.env` file. Record the model slug used and only non-secret result metadata in the area document before merge.

## Validation Record

- `pytest -q` passes with the OpenRouter client tests and no network access.
- A live request is intentionally pending: the operator must set a scoped `OPENROUTER_API_KEY` and choose the reviewed `VF_GPT_MODEL` slug.
