# v0.13.0 — D5 product traffic layer

**Status:** complete
**Owner / worktree:** Codex / repository-root `main`

## Problem and scope

D4's frozen data, trained checkpoints, held-out result, and M6 archive are
complete and must not be rerun. D5 needs a product-facing traffic source for
tomorrow's discovery/report work. This version adds only a standalone,
OpenAI-compatible proxy and a zero-cost synthetic traffic generator.

P1 creates `app/proxy/`, independently runnable with FastAPI. Its default
`VF_PROXY_UPSTREAM=fake` mode deterministically responds without network access.
`real` mode forwards the original chat-completions request to the canonical
`VF_LLM_BASE_URL` with `VF_LLM_API_KEY`, but this version does not invoke it.
Each request best-effort records timestamp, system-prompt SHA-256, model,
input/output tokens, latency, and configured-price cost in SQLite. A database
failure must never change the forwarding response.

P2 adds `scripts/traffic_gen.py`: 20 self-authored support-ticket examples, 20
invoice examples, and prompts sampled read-only from the frozen training pool.
It accepts `--rate`, `--total`, and weighted `--mix`, reports sent/success/fail,
and supports interruption. `scripts/traffic_summary.py` groups SQLite traffic by
system-prompt hash for D5 Discover's first data source.

## Explicit non-goals

No trainer, verifier, Storage, model, frozen data, existing API contract, mock,
frontend, deployment choice, real upstream call, or report-page change is in
scope. The SQLite database is local runtime data and never enters git.

## Affected areas and validation

Affected code: `app/proxy/`, `config/proxy_pricing.json`, `.gitignore`, and the
two traffic scripts plus focused tests. Contract impact is **none**: this is an
independent service with OpenAI-compatible request/response pass-through.

Validate P1 with fake-route shape, deterministic fake upstream, SQLite fields,
cost calculation, and non-blocking database failure tests. Validate P2's mix
construction and summary query, then run the independent proxy in fake mode and
send 300 mixed local requests. Acceptance requires 300 SQLite rows, three
distinct system hashes, a reasonable positive cost total, `pytest -q`, and a
commit/push after each P1/P2 wave.

## P1 result

P1 is implemented without a real upstream request. `app.proxy.main:app` starts
independently, defaults to fake mode, and records metadata through the
best-effort SQLite writer. `VF_PROXY_UPSTREAM=real` obtains only the canonical
`VF_LLM_*` configuration and forwards via the compatible endpoint; tests inject
that seam rather than contacting it. `config/proxy_pricing.json` supplies a
default rate and optional model override, while `traffic.db*` is ignored.
Focused proxy/client tests passed (**23 passed**).

## P2 result

The generator and summary scripts are complete. The acceptance replay used the
fake proxy only, at `http://127.0.0.1:8012` because port 8002 was already in
use by an unrelated local process. It sent **300/300** successful mixed
requests, persisted **300** rows with **3** distinct system-prompt hashes,
counted **26,650** total tokens, and estimated **$0.0077125** using the local
price table. The hash-group summary produced three 100-request clusters; no
real upstream request, D4 rerun, or frozen artifact mutation occurred.
