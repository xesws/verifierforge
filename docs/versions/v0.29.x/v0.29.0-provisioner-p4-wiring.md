# v0.29.0 — Provisioner P-4 product wiring

**Status:** P4-1/P4-2 complete; P4-3 live gate pending — 2026-07-19
**Owner/worktree:** Codex on `main`
**Release type:** minor product/provisioning capability

## Problem and scope

P-2 proved the separately authorized RunPod executor, but the product still
ends at an approval record. P-4 adds encrypted BYO provider credentials and an
explicit second confirmation that translates an approved `TrainingConfig` into
a bounded `ProvisionSpec`. The user-visible lifecycle is `approved →
provisioning → running → collecting → done` and remains pollable.

`Approve & Forge` continues to write only an approval. `Start Forge` is a
different endpoint and requires both an existing approval and an explicit
confirmation. Its budget is the lesser of the approved config cap and the
system cap. `VF_AUTOPROVISION` defaults to false and is the only switch allowed
to unlock provider mutation.

## Non-goals

Do not alter trainer logic or frozen data, enable autoprovision by default,
implement Nebius, cache decrypted keys, expose secrets in API responses, or
make any LLM request. P-4's one real smoke may provision and terminate one
minimal RunPod resource but need not train.

## Affected areas and contracts

- Settings API for write-only provider keys encrypted with `VF_CRED_KEY`.
- Credential resolution at the exact provider call boundary, with local demo
  fallback to `RUNPOD_API_KEY` when no user credential exists.
- Additive Start Forge request/status response contracts and DB-backed visible
  execution state; existing Agent, Approval and frontend fields stay intact.
- FastAPI-hosted Discover UI with a separate confirmation and status poll.
- CI secret scan and tests for encryption, redaction, flag-off behavior,
  idempotency, budget translation and mock full lifecycle.

## Validation and stop conditions

Run focused tests, full `pytest -q`, the secret scanner and shell/compile checks
before provider access. The live smoke has a `$1` cap and the wave retains its
`$5` cumulative fuse. Require approval, confirmation, real create, accepted
DELETE and fresh raw `vf-auto-* = 0`; any credential leak, wrong identity,
budget breach or cleanup failure blocks `provisioner-p4-complete`.
