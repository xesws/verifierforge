# v0.12.6 — Checkpoint space governance and M3 resume

**Status:** S1--S3 complete; S4 is blocked on an H100 cleanup check
**Owner / worktree:** Codex / repository-root `main`

## Problem and scope

v0.12.5 crossed the former epoch cap but its step-150 checkpoint publication
failed with `[Errno 122] Disk quota exceeded`. The wrapper stopped publishing
while the child continued, leaving only Storage steps 50/100 resumable. This
version safely recovers unneeded checkpoint copies, makes retention explicit,
preflights projected filesystem demand, and resumes the interrupted M3 from
Storage step 100 rather than restarting it.

## Fixed decisions

- Evidence is metrics, logs, runtime evidence, and a failed checkpoint tree;
  it is not every prior native/HF weight copy. Delete successful staging trees.
- Retain only the newest complete native checkpoint (model, optimizer, data)
  for resume. Retain every `actor/huggingface` export for M5, including exports
  under older checkpoint wrappers.
- On a successful bridge publication, delete its staging native directory. On
  a publication failure, preserve the failed native tree by atomically moving
  it into job evidence, record the error, and terminate the child rather than
  leaving an unbridged trainer running.
- Preflight uses `checkpoint_count = ceil(total_steps / checkpoint_every)` and
  checks the projected peak: `checkpoint_count * hf_export_bytes +
  3 * full_checkpoint_bytes` (all retained HF exports plus latest native,
  active staging, and atomic-copy peak). If this exceeds 80% of filesystem free
  space, it rejects before launching and prints every input and result. A
  resumable checkpoint supplies measured sizes; fresh runs use the cached model
  size with a conservative native multiplier.
- S2 deletes only the explicitly authorized old D4 checkpoints/staging and
  completed staging trees. The new job's failed step-150 tree is moved to
  evidence, not deleted. Frozen datasets, verifier, model weights cache, and
  old metrics/logs are non-goals.

## Affected areas and validation

`core/storage/local.py`, `trainer/checkpoint_bridge.py`, `trainer/grpo_train.py`,
focused Storage/bridge tests, run documentation, and remote operator commands
may change. Public contracts and frozen data have no impact. Matching
model-trainer, infrastructure, and evaluation-serving documents are updated
before code.

Validation completed locally: focused Storage/bridge/D4 tests passed (24), the
full suite passed (**200 passed, 1 skipped**), and shell syntax checks passed.
The tests cover retention of all HF exports with exactly one native resume
payload, rejection of an HF-only resume source, staging deletion on success,
atomic evidence preservation on failure, and capacity accept/reject arithmetic.
Remote S4 must still resume from exactly step 100 and clear the H100 before
M4--M6 may begin.

The first S4 preflight found tmux absent and the Storage step-100 native payload
intact, but `nvidia-smi` reported 2,134 MiB used while its compute table
identified only `PID 2743762, [Not Found], 2118 MiB`. The operator explicitly
accepted this ~2.6% host-namespace residual, prohibited a GPU reset, and
authorized resume. The guard window ends only when the resumed step-150
checkpoint is published with approximately 3.5 steps/minute and no CUDA/OOM
error; any anomaly kills the job and stops the run.
