replaying the completed Gate B workload exactly: the detached tag

`ad3d36b5860c66567a9f60f94c34101c787aaef3`, its original
`trainer/verl_configs/grpo_v1_0p5b.yaml`, and the frozen 50-row input.

`e2b8e973cf2400040dd83684e9bd7f4b0f191390ebce24ee43281f9ed5b81171`; it is
not the current file, which added fields after Gate B. A detached tag worktree

than reconstructing or editing a configuration. The existing H100 venv and
cache remain the runtime under test; only `py-spy` may be added to that venv.

## Execution and evidence

Start `c1-gateb-replay-v0123` in detached tmux with the tag worktree, the
original 0.5B config, and `VF_RUNS_DIR=/workspace/verifierforge/runs` so all
logs, metric artifacts, source-config copy, PID marker, and evidence live in
the normal persistent run directory. At T+3 minutes, install `py-spy`, select
the live `WorkerDict` process, and atomically write a full
`py-spy dump --native` (including the selected PID manifest and installer
output) under `runs/c1-gateb-replay-v0123/evidence/`. This capture is required
even if metrics have already appeared.

The total C1 timebox is 15 minutes. A JSONL metric row before expiry classifies
the H100/runtime as capable of the old workload and selects C2; no C2 repair
is authorized by this document. No metric by expiry selects C3 as an
environment-level regression candidate; it also makes no repair choice. In
either case W1 cleanup must remove tmux/processes and leave 0 MiB GPU memory.
