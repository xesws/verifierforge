# v0.12.5 — Total-step guard and fresh D4 rerun

**Status:** approved; documentation gate complete before implementation
**Owner / worktree:** Codex / repository-root `main`

## Problem and scope

The approved v0.12.4 M3 job requested 400 steps but stopped at 120 because
verl's 12-batch train loader exhausted a fixed `trainer.total_epochs=10` outer
loop first. v0.12.5 removes that hidden cap, preserves the stopped run as
evidence, and starts a new 400-step M3 job only after the guard and regression
coverage pass. A completed M3 automatically permits the already designed
serial M4, held-out-only M5, and M6 archive flow.

## Fixed decisions

- `total_epochs: null` means resolve at launch from the actual prepared input:
  `ceil(total_steps / steps_per_epoch) + 2`. `steps_per_epoch` is the exact
  drop-last batch count used by the prepared train file.
- A positive explicit `total_epochs` remains allowed, but a value below
  `ceil(total_steps / steps_per_epoch)` rejects the launch before verl starts.
  It is never silently allowed to truncate a requested step target.
- `grpo_v1_1p5b_h100_main` explicitly uses 40 epochs (at least 34 required by
  the observed 12 batches/epoch); the 0.5B control uses the safe derived
  default.
- The rerun uses a new job ID, no resume state, and preserves
  `d4-m3-1p5b-v0124` metrics and steps 50/100 unchanged as evidence.
- At 50 metrics, compare measured rate with the known 3.5 steps/minute. A
  rate outside 1.75--5.25 steps/minute is reported but does not stop training.
- M4 is serial, 0.5B, 200 steps, and deterministic random reward. M5 evaluates
  only the frozen 60-row held-out set at `k=8`; M6 archives curve, final
  checkpoint, hashes, and runtime evidence. An entropy brake, any gate refusal,
  or held-out after pass@1 below 0.5833333333333334 stops for human direction.

## Affected areas and validation

`trainer/grpo_config.py`, `trainer/grpo_dataset.py`, GRPO YAMLs, command
construction, focused trainer tests, and the P0 run sheet may change.
Matching model-trainer, infrastructure, and evaluation-serving documents are
updated before code. Public contracts, frozen data, verifier behavior, mock,
and frontend are non-goals and have no contract impact.

Validate focused guard/config tests, `pytest -q`, launch-shell syntax, a remote
import preflight, runtime evidence, new-job M3 completion at step 400 with a
final artifact, serial M4 completion, hash-bound M5 evidence, and M6 SHA-256
matches. Do not claim a result from the old partial checkpoints.

## Guard implementation result

The guard is implemented and locally validated. A 12-batch M3 input resolves
the explicit 40 epochs (minimum 34); the 200-step control resolves to 19
epochs (minimum 17). The old 10-epoch / 12-batch / 400-step combination now
raises before command launch. Focused tests, full `pytest -q` (`195 passed, 1
skipped`), and both shell syntax checks passed. Fresh remote M3 is the next
authorized stage.

## M3 Storage-gate stop

The fresh run reached 150 bridged metrics, passed engine initialization, and
published Storage checkpoints at steps 50 and 100. At the step-150 checkpoint,
`LocalStorage.save_checkpoint()` raised `shutil.Error` with `[Errno 122] Disk
quota exceeded`; no step-150 checkpoint or final artifact was published. The
bridge parent exited while the child verl process continued, so `vf kill` was
used to stop the process group and Ray. It left a GPU-cleanup gate failure:
`nvidia-smi` reports an unowned `[Not Found]` PID `2743762` at 2,118 MiB.

The partial M3 data is evidence only, not an eligible held-out candidate. M4,
M5, and M6 were not launched. The shared mount had 135 TiB free, but its
specific quota was not queryable (`quota` is unavailable); new-job `ckpt/` and
staging usage were 28 GiB and 41 GiB, respectively, and the old evidence run
remains 55 GiB. A separately approved repair plan is required before cleanup,
retention changes, or any new job.
