# v0.12.7 — Export/serving compatibility and M5 rerun

**Status:** planned; documentation gate complete before diagnosis or code
**Owner / worktree:** Codex / repository-root `main`

## Problem and scope

M3 completed and retained eight Hugging Face checkpoint exports, but the first
M5 evaluation server failed before readiness at step 50 with `ValueError: There
is no module or parameter named 'base_model' in Qwen2ForCausalLM`. The frozen
held-out identity passed; its unavailable report has `after: null`, so no gain
claim exists. This version diagnoses the export shape, creates a non-destructive
serveable representation for all eight exports, proves it with a real loopback
completion, reruns M5, and hardens future bridge publications with the same
serving-side acceptance definition.

## Fixed decisions

- X1 is read-only and timeboxed to ten minutes. It records the step-50 export
  file list, the first ten safetensors keys, and active LoRA/PEFT configuration.
  It must classify the export as pure adapter, `base_model`-prefixed full
  weights, or other. `other` stops for human direction.
- X2 keeps every original `actor/huggingface` directory unchanged. It writes a
  sibling `actor/serveable_huggingface` standard-HF directory for each of the
  eight steps. Pure
  adapters are merged into the frozen Qwen2.5-1.5B base in bf16; a confirmed
  key-prefix case is rewritten with a matching safetensors index. No training,
  verifier, frozen data, or original checkpoint is replaced or deleted.
- X3 must prove the converted step-50 directory through a pod-local vLLM
  `/v1/models` response and one real completion before M5 can start.
- X4 evaluates all eight converted checkpoints on exactly the immutable 60-row
  held-out set at `k=8`, storing every completion and tier score. It selects the
  highest held-out pass@1, lower step on ties. The sole automatic stop that
  requires human direction at this stage is best after pass@1 below
  `0.5833333333333334`; otherwise X5 proceeds automatically.
- X5 archives selected checkpoint, M3/M4 curves, hashes, runtime evidence, and
  adds a tested bridge publication gate: an export counts as published only if a
  local vLLM load smoke succeeds. This may not silently turn a failed serving
  smoke into a published checkpoint.

## X1 classification result

X1 classified the step-50 source as a `base_model`-prefixed full export with
PEFT wrapper state. It has no `adapter_config.json` or adapter-only weight
file; it has two `model-*.safetensors` shards and a standard Qwen2 config. Its
first keys are `base_model.model.model.embed_tokens.weight` and
`base_model.model.model.layers.0.mlp.down_proj.base_layer.weight`, followed by
rank-16 `lora_A.default` and `lora_B.default` tensors. The inventory is 280
base-layer keys, 196 A keys, 196 B keys, and 59 ordinary keys. M3 explicitly
configured rank 16, alpha 32, and `all-linear` target modules.

The human-specified prefix branch therefore applies, but the raw key evidence
requires one implementation refinement: a prefix-only rewrite would retain
unloadable `.base_layer` and LoRA keys and lose the trained delta. Codex will
instantiate the matching PEFT wrapper, load the complete source state, call
`merge_and_unload`, and save a standard bf16 sibling directory. This is a
documented implementation choice within X2, not a change to data, training, or
the selected conversion outcome.

X2 completed at 2026-07-17 04:50:18 UTC for all eight steps 50--400. Each has
an atomic `actor/serveable_huggingface` sibling and manifest; the source exports
remain present. The batch summary SHA-256 is
`59f36ea655e8106be87b152b46359363a1c7368091eec5ff981992c22b373ddb`.
The standard bf16 step-50 result is 3.4 GiB. Local converter/held-out focused
tests passed (27) and the full suite passed (**205 passed, 1 skipped**) before
the pod batch began. X3 is now the only admission gate before M5 reruns.

## Explicit non-goals

No new training run, model hyperparameter change, dataset/verifier change,
external API request, or deletion of original exports is allowed. A converted
representation is a serving artifact, not a replacement training resume source.

## Affected areas and validation

Expected implementation areas are `trainer/heldout_eval.py`, a small export
compatibility/serving-smoke module, bridge publication integration, and focused
pytest coverage. Matching model-trainer, evaluation-serving, and infrastructure
documents plus `docs/p0-run-sheet.md` are updated before code. Validate X1
evidence, converter tests, no mutation of originals, local vLLM models plus
completion, M5's 8 × 480 sample evidence, report selection/tie-break, and M6
hash parity. Any converter, serving, evidence, or hash failure stops without a
gain claim.
