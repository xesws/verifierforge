# v0.12.1 — Provisioning completion semantics

**Status:** planned; documentation gate complete before implementation
**Owner / worktree:** Codex / repository-root `main`
**Predecessor:** v0.12.0 N3 stopped after a clone/venv existed but the required
runtime packages were absent.

## Problem and scope

The fresh H100 N3 command returned after cloning, but the required Python
runtime (`torch`, `vllm`, `verl`, `ray`) was not present. Read-only evidence
found no tmux server, no `runs/provisioning/` directory or runtime-install log,
and no installed target package. `trainer/bootstrap.sh` already executes its
two pip lines synchronously under `set -euo pipefail`, with no `|| true` on the
installation path; however, the `vf bootstrap` transport had no durable status
or log boundary, so it could not prove completion or preserve failure output.

v0.12.1 makes the bootstrap lifecycle observable and truthful: the remote
installation runs in named tmux session `vf-bootstrap`, appends to
`runs/provisioning/runtime-install.log`, writes an explicit completion-status
artifact, and `vf bootstrap` waits synchronously for that status before it may
return success. If the SSH caller disconnects, tmux continues; no subsequent
command may claim bootstrap completion until the status says success.

## Non-goals and stop conditions

- Do not change the pinned requirements or accept substitute versions. N3 still
  requires exactly torch `2.8.0+cu128`, vLLM `0.10.2`, verl `0.8.0`, and ray
  `2.56.0` after import. Any mismatch is a hard stop.
- Do not start N4–N7, copy `.env`, install unrelated packages, or modify
  training logic/configuration in this patch.
- The explicit post-fix install may run in the new tmux session and retain its
  log. A nonzero status, missing completion artifact, interrupted tmux session,
  or import/version failure stops immediately and preserves the raw log.

## Affected areas and validation

- `trainer/bootstrap.sh` obtains log/trap status publication without masking
  pip/apt failure.
- `scripts/vf` starts/observes the named session and returns zero only after a
  successful status artifact; it retains the root-only workspace-key copy path.
- Focused control-plane tests verify the session, log, and failure/success
  sentinels; run `bash -n scripts/vf`, `pytest -q`, then the live H100 N3
  provisioning/import/version acceptance. Record every outcome in
  `docs/p0-run-sheet.md`.
