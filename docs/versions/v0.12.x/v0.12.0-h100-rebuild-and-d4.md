## Problem and scope

A fresh, single-H100 SXM 80 GB RunPod executor replaces the stopped Blackwell
pod. The workflow starts from a newly empty `/workspace`, establishes its own
read-only deploy key, installs the pinned trainer runtime, downloads only the
two Qwen 2.5 models, and verifies the existing immutable v0.10.2 three-piece
freeze. Only after those gates may it run a bounded H100 M2′ smoke.

The H100 smoke has one GPU, rollout TP 1, GPU memory utilization `0.45`, and
the existing Hugging Face `sdpa` attention override. It uses a new H100-named
configuration rather than mutating historical Blackwell evidence. N7 begins
only if N6 publishes at least twenty complete metric rows and W1 teardown
returns the GPU to zero; it then launches the approved M3–M6 sequence.

## Decisions, non-goals, and stop conditions

- The laptop remains the control plane; the H100 is a disposable executor.
  Persistent pod state belongs under `/workspace`; the pod deploy key is
  restored as a root-only `0600` copy before Git operations. No laptop `.env`
  or API key is copied or printed.
- N1 through N6 are strict sequential gates. A failed SSH/GPU/volume check,
  unauthenticated GitHub CLI, bootstrap import/version mismatch, model snapshot
  failure, tag/hash mismatch, or insufficient smoke evidence stops immediately.
  No later step is skipped or started.
- The only model downloads are `Qwen/Qwen2.5-0.5B-Instruct` and
  `Qwen/Qwen2.5-1.5B-Instruct`; `openai/gpt-oss-20b` is explicitly excluded.
- No claim may use the training pool for before/after quality. The frozen
  60-row held-out set remains the sole D4/DoD measurement source. N7 must keep
  raw evidence, Storage artifacts, and synchronized SHA-256 results.

## N6 implementation choice

The implementation adds `trainer/verl_configs/grpo_v1_1p5b_h100_smoke.yaml`
and its explicit `trainer/launch.sh` target. It is intentionally distinct from
the historical Blackwell configuration: 30 total steps, 1.5B model, `k=8`,
one GPU/TP 1 (the existing command builder's fixed declarations), rollout GPU
memory utilization `0.45`, and Hugging Face `sdpa`. `enforce_eager=false`
allows H100's normal V1 path; `vllm_attention_backend=null` ensures no legacy
`VLLM_ATTENTION_BACKEND` setting forces an incompatible V0 fallback. Focused
tests must bind these exact facts before a pod pull/launch.

**Implementation validation:** the target, config, and focused assertions now
exist; `bash -n trainer/launch.sh` and the full suite passed (`185 passed,
1 skipped`) before the H100 pull.
