# v0.25.0 — Supabase landing and historical import (DB-2)

**Status:** complete — 2026-07-18
**Owner/worktree:** delegated migration area, integrated by Codex on `main`
**Release type:** minor database deployment capability

## Problem and scope

DB-2 applies DB-1's Alembic schema to Supabase, imports every existing
relational row plus local job summaries, proves idempotence and content/count
parity, and runs the repository/API suite against true Postgres. The source
SQLite file is opened read-only from an immutable snapshot; local `runs/` logs
are never copied into the database.

Application code requires `SUPABASE_DB_URL`. The direct endpoint was unreachable
from the IPv4-only development host, so the owner supplied the Supavisor session
pooler DSN through the ignored `.env`; no URL or password was printed.

## Non-goals

Do not change the code-level SQLite fallback, delete/archive the source SQLite
file, write provider credentials, modify S3 objects, call a GPU/LLM/vendor API,
or add a dual-write period. The owner alone authorizes the persistent
environment switch.

## Affected areas and interfaces

- Add a read-only SQLite/runs importer with deterministic source identities,
  dry-run/apply/verify modes, all-or-nothing transactions, and sanitized errors.
- Seed the shared three-cluster catalog and migrate traffic, routing, guardian
  scores/points, jobs, Agent decisions, and approvals.
- Run one parameterized repository suite against Postgres using an outer
  rollback transaction so tests do not pollute imported history.
- Preserve explicit local fixture isolation after the persistent Postgres
  switch: a caller-supplied Agent `db_path` must select that SQLite file and
  must never be redirected to the process-wide Supabase backend.

No API/Pydantic response field changes.

## Validation plan

Freeze source SHA/counts; migrate schema; import twice; require the second pass
to insert zero rows; compare every table's count and canonical content digest;
run Postgres repository tests, the complete suite, and a process-local product
smoke. A permission/403 error is attempted once and becomes `OWNER-ACTION`.
`db-2-complete` is withheld until the owner persists `VF_DB_BACKEND=postgres`
and the full product regression passes in that configuration.

## Result

The read-only planner and idempotent importer are complete. The current source
plan is 220 traffic rows, 3 clusters, 1 route, 113 guardian scores, 113 live
points, 16 jobs, 48 Agent decisions, and 1 approval. Local
apply/reapply/verify tests pass.

The initial direct-endpoint attempt stopped before DDL with:

```text
[Errno 8] nodename nor servname provided, or not known
```

The replacement session-pooler connection passed `SELECT 1`. Alembic revision
`20260718_0001` was applied, the PostgreSQL repository suite passed `4 passed`,
and the second import inserted zero rows. All eight expected/actual counts and
canonical digests matched. The six-step product smoke passed against Supabase,
including proxy traffic, clusters, routing, guardian points, Discover, Agent
decisions, and approvals.

The persistent Postgres regression first found three Agent fixture-isolation
failures: explicit SQLite `db_path` values were incorrectly overridden by the
global backend. Codex classified this as a configuration-precedence bug, made
explicit paths authoritative, added a non-connection regression, and reran the
suite successfully: `365 passed, 1 skipped`. The owner-authorized ignored
`.env` now selects `VF_DB_BACKEND=postgres`.
