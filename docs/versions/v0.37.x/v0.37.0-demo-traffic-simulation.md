# v0.37.0 — one-click demo traffic simulation

**Status:** complete; deployed and publicly accepted on 2026-07-20

## Problem and scope

The public Ship page can configure production canary routing and display live
Guardian points, but reviewers still need an operator-run script to create the
traffic that makes those controls visible. This version adds an authenticated,
feature-flagged demo task that reuses the existing mixed `traffic_gen` module
and the exact in-process proxy pipeline, plus a Ship-page button with live
progress and automatic Guardian refresh.

The task accepts at most 500 requests, defaults to 200 requests at 5 requests
per second, and admits one process-local run at a time. A duplicate start
returns the existing run status. Ready registry endpoints follow the saved
canary policy; cold endpoints keep the established successful physical-default
fallback behavior and never schedule Guardian scoring.

## Explicit non-goals

- No subprocess, loopback self-request, new proxy route, or alternate routing
  implementation.
- No database migration, durable task queue, cancellation endpoint, multi-worker
  coordination, `core/` contract change, paid evaluator, or model change.
- No change to the frozen 22-operation frontend API v1 or mock-server parity;
  `/demo/*` is an additive reviewer-only extension.
- No change to the observable `default-fallback` label for cold tuned draws.

## Contract impact

Additive reviewer extension only:

- `POST /demo/traffic` accepts `total=200` (`1..500`) and `rate=5.0`
  (`0 < rate <= 20`). A new run returns 202; a duplicate while running returns
  200 with the active run.
- `GET /demo/traffic/status` returns `total`, `rate`, `sent`, `success`,
  `failed`, `running`, and a nullable sanitized `error`.

Both routes require the existing invitation and return 404 while
`VF_DEMO_TRAFFIC_ENABLED=false`. The frontend sends `{total: 200, rate: 5}`.

## Public acceptance result

Railway deployed exact implementation SHA `4f747e5` with one replica. With the
registry cold, the default 200-request run completed 200/200 with zero failure:
134 rows were labeled `default`, 66 SQL canary attempts were observably labeled
`default-fallback`, and all 200 physically used the default upstream. Guardian
remained at 143 points with zero score delta.

After one authorized Wake and a saved 50% SQL canary, the second default run
completed 200/200 with 159 `default` and 41 `tuned` rows. Guardian grew live
from 143 to 145 points; the first new point was visible from the public API at
46/200 while the task was still running. The original enabled 100% route and
30-minute idle policy were restored. The endpoint returned to cold with
provider deletion confirmed, and a raw RunPod inventory read found zero
`vf-auto-*` resources.

Vercel deployment `dpl_5PAUihiKB6uncDadX3dpRPKFdB94` is Ready at the stable
`https://verifierforge-web.vercel.app` alias. The public Ship route, deployed
control strings, exact-origin CORS GET/POST preflight, and no-refresh page-flow
test pass. No in-app browser instance was available, so this record makes no
production screenshot or manual visual-inspection claim.
