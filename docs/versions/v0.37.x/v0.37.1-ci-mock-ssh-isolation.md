# v0.37.1 — isolate mock provisioning from host SSH state

**Status:** implemented and locally validated; pushed CI confirmation pending

## Problem and scope

The GitHub `security` workflow runs the P4 mock lifecycle on a clean Python
3.11 runner. `test_approval_then_separate_start_runs_complete_mock_lifecycle`
does not provide `VF_PROVISION_SSH_PUBLIC_KEY`, so the mock path falls through
to the operator-only `~/.ssh/id_ed25519.pub` default. Developer machines with
that file pass; clean CI runners fail the otherwise zero-cost mock lifecycle.

This patch makes the test fixture explicitly self-contained by injecting a
synthetic, non-secret SSH public key through the existing environment seam.

## Explicit non-goals

- No production provisioning, RunPod, serving, API, proxy, frontend, database,
  or contract behavior change.
- No relaxation of the real RunPod public-key requirement.
- No retry, skip, timing sleep, CI exception, or workflow weakening.
- No modification of the v0.37.0 traffic simulation acceptance evidence.

## Affected areas and ownership

- `tests/test_p4_api.py`: isolate the mock lifecycle fixture from host files.
- `docs/infrastructure/`: record the clean-runner failure and repair.
- `docs/p0-run-sheet.md`: track the patch gate and final CI result.
- Owner/worktree: primary Codex integration agent on `main`.

## Contract impact

None. The test continues to exercise the same API request and complete mock
lifecycle. The injected value is a public test fixture and never reaches the
live provider binding.

## Validation plan

1. Reproduce the clean-runner condition locally with an explicit nonexistent

   change.
2. Run the exact GitHub security pytest selection, full `pytest -q`, secret
   scan, Python compilation, and `git diff --check`.
3. Commit and push `v0.37.1`, then require the GitHub `security` workflow to
   pass. Confirm the automatic Railway docs/test-only deployment remains
