# v0.19.0 — Forge Agent contracts and read-only tools

**Status:** completed
**Owner/worktree:** Codex, repository `main`  
**Stage:** Forge Agent A-1

## Problem and scope

Define strict, auditable contracts for decisions, proposed training configs,
tool calls, and traces. Implement four deterministic read-only tools behind a
`real`/`mock` registry with identical Pydantic input/output types. Dependency
IDs bind later tools to actual earlier outputs so Gate C can measure real data
flow rather than matching a repeated cluster ID.

The only trainable base model is `Qwen/Qwen2.5-1.5B-Instruct`. The owner-set
training budget ceiling is $100; runtime policy may lower but never raise it.
Current traffic SQLite stores metadata, not request/response bodies, so the real
sample tool must report insufficient data rather than synthesize content.

## Non-goals

No LLM loop, persistence, evaluator, API/UI, approval, provisioner, GPU, or
training action. Tools do not mutate the traffic database or expand its privacy
boundary.

## Contract impact

New contracts live in `core/agent_contracts.py`; existing Job/Cluster models are
unchanged. Strict schemas reject unknown fields, NaN values, invalid action/
config combinations, and training budgets above the owner ceiling.

## Validation plan

Protect JSON Schema, boundary validation, decision/config coupling, deterministic
IDs, dependency rejection, database read-only behavior, real/mock output-schema
identity, economics formulas, and the explicit insufficient-sample path.

## Result

The contracts and both registry bindings are implemented. Focused validation
passed 17 tests; the complete repository suite passed 272 tests with one
expected skip. A byte comparison around the real SQLite chain confirmed that
analysis/economics reads do not mutate the database, and a missing database was
not created. The real sample path truthfully returned insufficient data.
