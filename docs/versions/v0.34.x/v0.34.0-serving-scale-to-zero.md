# v0.34.0 Scale-to-zero serving

**Status:** planned; documentation gate open — 2026-07-20

## Problem and scope

VerifierForge currently routes tuned traffic to one manually maintained L4.
This wave adds a disabled-by-default, invite-protected serving session that
uses the existing capacity-aware RunPod adapter to provision a small Ada GPU,
materialize the frozen step-350 model from S3, start vLLM, publish a dynamic
endpoint, and terminate it after an idle interval.

The canonical model is `vf-demo`, sourced from the 13-file step-350
`serveable_huggingface` tree whose canonical SHA-256 is
`7bde853af7c82405fd1356de9bad9b6c421de45a45ce747f63ea2f8a27eda658`.
The implementation must preserve static report/arena fallback whenever no
ready endpoint exists.

## Non-goals

- Do not stop, restart, or reconfigure the existing L4 before the full SV gate.
- Do not enable training autoprovisioning or reuse the Start Forge boundary.
- Do not change frozen training data, trainer logic, or existing response fields.
- Do not add a second serving model, Nebius adapter, paid LLM call, or domain.

## Affected areas and ownership

- Contracts: additive serving state/request/status models.
- Database: endpoint registry, append-only serving audit, encrypted key reference.
- Provisioner: RunPod port/start-command support and a recoverable serving driver.
- API/proxy/frontend: wake/status, dynamic resolution, safe fallback, reviewer UI.
- Infrastructure/evidence: S3 model publication, hosted secrets, two live wake cycles.

Owner/worktree: primary Codex agent on `/Users/tangyiq/dev/verifierforge` `main`.
Target commits use the `v0.34.0` prefix; no parallel worktree is authorized.

## Contract impact

The change is additive. `POST /serving/wake` and `GET /serving/status` join the
frozen frontend surface; real and mock APIs must remain shape-identical. Public
routing continues to expose only the logical `tuned` alias. Physical URLs and
credential references remain private repository data.

## Validation plan

Run focused contract, migration/store, adapter, session, proxy, API, reviewer,
mock-parity and UI tests, then `pytest -q` and the secret scanner. Live
acceptance uses one active session maximum, a `$5` per-session cap, a 120-minute
runtime cap, production idle timeout 30 minutes, and a 3-minute acceptance
override. The semantic tag `serving-scale-to-zero` is created only after two
wake/ready/terminate cycles and provider-side deletion proof.
