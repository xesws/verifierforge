# v0.17.0 — reproducible delivery artifacts

**Status:** planned
**Owner / worktree:** Codex / repository-root `main`

## Problem and scope

The D4 result exists in ignored run storage, which is correct for checkpoints
but unsuitable for a reviewer who clones the repository without the original
pod. This version promotes a small, reviewed subset of factual D4 evidence into
`data/demo-artifacts/` and adds an explicit read-only API artifacts mode.

It also completes the delivery surface: a truthful README, an external-facing
development narrative, synchronized contributor rules, a locked serving
requirements file, submission prose/video plan/judge path, and a clean tracked
working tree. The D4 held-out result remains the factual
`0.5833 → 0.7833` pass@1 comparison; no new training or evaluation is run.

## Explicit non-goals

No `trainer/` change, frozen-data change, contract-field change, paid model
call, public endpoint deployment retry, or claim that the blocked RunPod proxy
is available is allowed. Demo artifacts contain metrics, reports, and hashes,
never checkpoint weights, `.env`, credentials, or raw private traffic.

## Affected areas and validation

Affected implementation is `app/api/` plus `data/demo-artifacts/` and API
tests. `VF_API_DATA_MODE=artifacts` selects shipped, read-only JSON/JSONL data;
the normal local-runs mode remains default. Job/metrics/routing/live-pass
responses must validate against the existing contracts, and artifact metrics
must match their committed JSONL sources exactly. Artifact-mode PUT routing is
explicitly read-only.

Documentation changes cover frontend consumption, backend behavior,
evaluation provenance, serving operations, README, `docs/dev_doc_v0.md`, and
`AGENTS.md`. Validation includes focused API consistency tests, `pytest -q`,
and a static artifact-mode API smoke. Submission/privacy actions remain
separately recorded because GitHub permission can block them without blocking
the repository deliverable.
