# v0.28.3 — P-2 chained retry and remote failure detection

The final P-2 attempt uses `-r3` and is admitted only from the immutable `-r2`
lifecycle. The original approval remains bound to the first terminal handle;
each replacement is linked through explicit audit/evidence rather than
rewriting that binding. The prior conservative reservation is carried forward
and the second attempt's elapsed wall time is added at the maximum permitted
wave burn rate before any create.

The S3 controller must treat the existing checkpoint-publication-failure
manifest as a terminal remote signal. This closes the gap where detached tmux
had exited but the controller could otherwise poll incomplete metrics until
the runtime fuse. Cleanup still requires DELETE, target absence and raw
`vf-auto-*` prefix zero.
