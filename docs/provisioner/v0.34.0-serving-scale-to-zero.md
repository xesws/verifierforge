# v0.34.0 Serving session provisioner

**Status:** implemented and mock-verified; live RunPod acceptance pending

Serving is a recoverable variant of the P-2 lifecycle, not a Forge training
approval. Durable state is `cold -> provisioning -> loading -> ready ->
draining -> cold`. Provider handles, timestamps, selected capacity, cost, and
sanitized failures are stored before each external transition so a control
plane restart can reconcile rather than orphan a billing pod.

RunPod creation must honor `ProvisionSpec.ports` as `22/tcp` plus `8000/http`
and fixed, secret-free container startup arguments. Capacity remains the
approved `small_ada` list with Blackwell blocked. Authentication/schema errors
fail immediately; only explicit capacity failures advance to the next model.

Ready requires the full chain: S3 per-file and canonical tree verification,
vLLM `/v1/models`, one real completion, cloudflared URL validation, then an
atomic registry transition. Idle or budget/runtime termination writes draining
first, performs DELETE, proves the target absent and the managed inventory
clean, then writes cold and schedules billing reconciliation.

The pod receives expiring GET URLs for the 13 manifest-published model objects,
one expiring PUT URL for its tunnel callback, and a per-session vLLM key. The
callback proves the canonical tree hash before the controller accepts the
tunnel. RunPod capacity fallback, ownership markers, DELETE readback,
managed-inventory-zero proof, and delayed billing schedule are reused.
