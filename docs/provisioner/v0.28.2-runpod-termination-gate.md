# v0.28.2 — RunPod termination gate

P-2 now separates resource safety from delayed accounting. A paid stage may
advance only after its DELETE call returns normally, the target reads absent,
and the raw account pod list contains no name beginning `vf-auto-`. This last
check deliberately does not use `list_active()`: terminal or malformed
prefix-owned resources must still block the next create.

The prior gold pod is admitted from matching immutable evidence plus a fresh
provider readback. Resume must not overwrite the failed v0.28.0 lifecycle or
call provision for another gold pod. Orphan reaping and the 0.5B/100-step run
remain live P-2 gates.

Actual billing is `pending` until RunPod returns records. The budget fuse uses
conservative provider runtime estimates meanwhile; pending never means zero.

Capacity fallback is narrow: create first requests Community; only HTTP 500
whose provider body says `There are no instances currently available` may retry
the same ordered GPU list once in Secure. Any other provider error is returned
unchanged. The fallback remains non-interruptible and inside the existing cost
and runtime fuses.

A failed full attempt may resume only when its lifecycle shows the prior gold
and orphan pass, a training create followed by confirmed deletion, the bound
approval still names that exact terminal handle, and live raw prefix count is
zero. Resume skips gold/orphan, creates a clean retry job ID, leaves the exact-
once approval binding untouched, and records the replacement handle through
audit/evidence. Failed-attempt wall time is charged conservatively against the
same `$5` wave before the retry create.
