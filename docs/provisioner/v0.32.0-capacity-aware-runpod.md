# v0.32.0 — RunPod capacity-aware selection

The RunPod adapter will treat each `gpu_class` mapping as an approved candidate
set. Immediately before provider creation it queries official GraphQL
`gpuTypes`, resolves the lowest on-demand Community/Secure offer per model,
filters unavailable/blocked entries and sorts offers by hourly price with the
declared candidate order as the deterministic tie-breaker.

Each create request contains exactly one `gpuTypeIds` value and the selected
cloud type. Provider error or create timeout advances once to the next ranked
model. Exhaustion raises `no_capacity`; the orchestrator appends a FAILED
audit and returns without a handle. On success `provision.created` carries the
selected `gpu_model`, `cloud_type` and `hourly_price_usd` from the handle's
whitelisted labels.

Validation covers first-choice unavailable, all unavailable, only the third
candidate available, a capacity race during create, bounded attempts, audit
metadata and terminal cleanup. Live validation is one immediate-delete Pod,
not a training run.
