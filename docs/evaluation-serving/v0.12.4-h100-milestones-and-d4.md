# v0.12.4 — Evaluation after H100 D4 training

M5 is the exclusive before/after measurement step. It evaluates the immutable
60-row held-out set using the verifier and preserves model/checkpoint identity,
full sample evidence, the three metrics, and hashes beside the fixed before
snapshot `0.5833333333333334 / 0.7666666666666667 / 0.4666666666666667`.

Checkpoint selection is held-out-only: maximum held-out pass@1 wins and lower
checkpoint step breaks ties. If a checkpoint cannot be evaluated or represented
faithfully by the evaluator, the report must say unavailable and no best-model
or gain claim may be emitted. Training-pool metrics and M4's random-reward
curve remain monitoring/control evidence only.

M5 evaluates each eligible M3 checkpoint only after training releases the GPU.
The checkpoint's exported Hugging Face directory is served on pod-local
loopback by vLLM; the existing OpenAI-compatible evaluator receives only
ephemeral local `VF_EVAL_BASE_URL`, `VF_EVAL_MODEL`, and non-secret local API
key values. Each 60-row, `k=8` run retains 480 full completions with verifier
tier breakdowns, a hash-bound evidence report, and the three metric values.
Server startup, evaluation, and shutdown are one checkpoint at a time. Any
failure preserves its evidence and yields an unavailable report rather than a
substitute score.

**Implementation:** `trainer.heldout_eval` discovers only Storage checkpoints
that contain an exported HF safetensors model, hashes each export, starts vLLM
on `127.0.0.1`, and invokes `scripts.gate_a` in reference mode with explicit
local-only evaluation variables. It validates the frozen input digest, 60×8
sample count, sample hash, and verifier version before selection. Any missing
export or failed result writes a hash-bound `unavailable` report instead of
choosing among partial results. Local unit coverage exercises checkpoint
eligibility, evidence rejection, and the lower-step tie-break.
