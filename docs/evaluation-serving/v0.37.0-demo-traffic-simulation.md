# v0.37.0 demo traffic routing and Guardian behavior

The demo replays the existing equal-weight support-ticket, invoice, and
data-pull-sql families. Their established system prompts enter the normal
cluster lookup, repository routing state, canary draw, registry resolution,
traffic persistence, and asynchronous Guardian path.

When the registry is ready, only requests selected by the saved route use the
registered tuned endpoint. Guardian still samples only successful tuned
`data-pull-sql` completions. When the registry is cold, all requests complete
through the physical default upstream; a canary-selected tuned attempt retains
the existing observable `default-fallback` path and schedules no Guardian
score. Cold acceptance therefore requires zero tuned rows, zero request
failures, and unchanged Guardian point count rather than relabeling fallback.

No evaluator, threshold, frozen label, prompt pool, model ID, or sampling rate
changes in this version.

The complete proxy regression suite passes alongside an in-process dispatcher
test; established cold fallback, canary, persistence, and Guardian scheduling
tests remain green.
