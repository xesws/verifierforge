# v0.12.6 — Preserve M5 checkpoint candidates

Space governance retains each successful checkpoint's Hugging Face export even
when its native optimizer/data payload is pruned. M5's existing discovery path
continues to use `global_step_<n>/actor/huggingface`, so every scheduled M3
checkpoint remains a held-out evaluation candidate.

The failed v0.12.5 step-150 tree is evidence, not a completed Storage export;
it is never a candidate. Resumed M3 has now reached step 400, published its
final artifact, and retained the complete 50--400 HF candidate set. M5 remains
blocked only on the serial M4 control completing successfully.

M4 is now complete, so M5 may run pod-local only. It serves each of the eight
M3 HF exports sequentially on loopback ports 8011--8018 and invokes the shared
evaluator on all 60 held-out rows with `k=8` and `--save-samples`. Any missing
480-sample checkpoint evidence, frozen identity mismatch, unavailable vLLM
server, or evaluator error publishes an unavailable report and prevents a
before/after claim.

The retention tests also prove that a higher HF-only wrapper is skipped by the
resume path while remaining discoverable by the held-out evaluator. That keeps
M5's candidate set intact without allowing an evaluator-only export to change
training state.
