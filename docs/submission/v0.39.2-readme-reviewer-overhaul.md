## Audience and message

The primary reader is a DevPost reviewer who should understand the business
value and reach the live demo without cloning the repository. The opening
message is: VerifierForge reduces repetitive AI spend by finding work that can
be checked automatically, then training, validating, and deploying a smaller
model to replace the expensive general-purpose call.

The reviewer receives the invitation code separately through DevPost, opens
`https://verifierforge-web.vercel.app`, and follows the enforced Discover →
Forge → Runs → Proof → Ship journey. The code remains out of Git, the URL, and
the frontend build. Vercel is the product UI; Railway is its API and control
plane, not a second site the reviewer needs to open.

## Reviewer truth boundaries

- Review requires no local setup. The clone-only sandbox is a fallback for
  contributors and judges who want to inspect committed artifacts offline.
- Hosted Analyze may use the configured live Agent, and Ship Wake may allocate
  one budget-capped GPU. These are not described as zero-cost operations.
- `Approve & Forge` records intent; public `Start Forge` remains disabled.
- Reports, curves, arena evidence, and browser-local SQLite execution remain
  usable while tuned serving is cold.
- Reviewers should use **Leave session** for confirmed provider deletion; a
  closed tab relies on the idle reaper instead.
