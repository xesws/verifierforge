# v0.12.0 — Trainer: H100 smoke and D4 progression

N6 may create an H100-specific 1.5B smoke configuration only after N1–N5 pass.
It preserves the frozen 50-row runtime fixture, one GPU, TP 1, `k=8`, Hugging
Face `sdpa`, and `enforce_eager`; it changes only rollout GPU memory utilization
to `0.45` for the H100 80 GB card. The bounded smoke is 20–30 steps, must
publish at least twenty nonempty reward/pass metric rows, and is killed through
W1 once the curve evidence is sufficient. Tokens/sec, step time, and the M3
runtime estimate are recorded in the run sheet.

N7 is gated strictly on that passing result. M3 uses the 1.5B model, 300–500
steps, `k=8`, 50-step checkpoints, an entropy brake, and held-out-only best
checkpoint selection. M4 is a serial 0.5B random-reward control at half the
M3 step count. M5 evaluates the frozen 60-row held-out set and compares its
three metrics only with the archived baseline `0.583 / 0.767 / 0.467`.

Before any implementation for M3–M6, this document and the version/run sheet
must name the exact new config, report/evaluator artifact, and validation. A
missing capability is a gate failure—not permission to weaken the held-out,
checkpoint, entropy, or evidence requirements.
