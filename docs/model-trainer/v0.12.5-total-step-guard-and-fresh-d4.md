# v0.12.5 — Trainer total-step guard and fresh D4 run

The launch boundary must resolve `trainer.total_epochs` only after it knows the
prepared train file's effective drop-last batches per epoch. A null YAML value
means `ceil(total_steps / steps_per_epoch) + 2`; a positive explicit value is
accepted only when it can reach the requested target. Command construction
receives the resolved value, so no GRPO command can retain the former hidden
constant `trainer.total_epochs=10`.

The main H100 config explicitly sets 40 epochs. With the observed 12 batches
per epoch, its 400-step target needs 34 epochs and has six epochs of margin.
The M4 200-step control keeps a null value and derives its own safe count.
Regression tests must cover derived resolution, an insufficient explicit value
failing before launch, and a valid explicit value appearing in the verl
command.

M3 is a fresh `d4-m3-1p5b-r1-v0125` job with no resume path. The prior
`d4-m3-1p5b-v0124` metrics and step-50/100 checkpoints remain immutable
evidence. After a complete 400-step M3 and final artifact, M4 starts serially;
M5 and M6 follow only under their existing held-out/evidence gates.
