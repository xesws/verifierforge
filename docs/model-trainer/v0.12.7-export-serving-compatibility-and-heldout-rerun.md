# v0.12.7 — Serveable checkpoint exports

M3's eight retained `actor/huggingface` exports are held-out candidates, but
they are not assumed to be directly loadable. X1 must first establish whether
they are PEFT adapters, prefix-shaped full weights, or another layout. The
classification determines the converter; no heuristic key rewrite is permitted
before evidence identifies the layout.

For a confirmed adapter export, the converter loads the immutable local
Qwen2.5-1.5B base, attaches the adapter, calls `merge_and_unload`, and writes a
bf16 standard Hugging Face directory beside the source. For a confirmed prefix
layout, it rewrites only the documented key prefix and safetensors index. Both
paths leave model/optimizer/data resume payloads and source HF files unchanged.

The bridge's future publication contract gains a serving acceptance gate. A
successful filesystem copy is necessary but insufficient: the candidate must
also be loadable by a loopback vLLM smoke. Tests cover conversion selection,
source immutability, bridge refusal on a failed smoke, and publication only
after a passed smoke. This changes checkpoint artifact semantics, not trainer
optimization or frozen-data inputs.
