# v0.11.0 — Trainer: D4 multi-GPU main/control runs

The Gate B configuration is not reused as a main run: it hard-codes one GPU,
0.5B, and 100 steps. D4 adds operator-visible configurations for a 30-step
1.5B multi-GPU smoke, a 400-step 1.5B main run, and a 200-step 0.5B random
reward control. Each preserves the frozen 50-row training-pool alias and
`rollout_n=8`.

The main process receives an explicit reserved-GPU count and uses FSDP plus a
matching vLLM tensor-parallel allocation. M2 must show both subsystems and a
post-update rollout before it is admitted. Main checkpoints publish through
Storage every 50 steps. An entropy-brake monitor observes append-only bridged
metrics and stops only after ten consecutive values below 25% of the initial
ten-step median, never before step 20.

The control uses a deterministic seed-derived Bernoulli(0.5) reward function;
it does not call the NL2SQL verifier. It runs on a non-overlapping GPU and has
its own job ID, checkpoints, and metrics. The trainer may not select a main
checkpoint using its 40/10 training split: only the external frozen held-out
evaluation may mark a checkpoint as retained. Tests cover configuration GPU
overrides, random-reward determinism, entropy-brake boundaries, and checkpoint
selection artifacts before any pod launch.
