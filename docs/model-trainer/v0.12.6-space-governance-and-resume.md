# v0.12.6 — Checkpoint retention and resumable M3

The bridge will publish each completed native checkpoint through Storage, then
retain only the newest complete native payload for resume. For every older
checkpoint it retains `global_step_<n>/actor/huggingface` and removes native
optimizer/model/data payloads, so M5 can evaluate every scheduled export.

Before a child launches, the trainer asks the bridge/storage budget helper for
the effective checkpoint count and peak bytes. A known resume checkpoint gives
measured native and HF sizes; a fresh run falls back to cached model bytes with
a conservative native estimate. The helper rejects projected demand above 80%
of free filesystem space and prints the calculation.

The v0.12.5 M3 wrapper failed while its child ran on. v0.12.6 treats bridge
publication failure as terminal: it records the error, preserves the failed
staging tree under evidence, terminates the child process group, and returns a
nonzero result. After S2 cleanup, `d4-m3-1p5b-r1-v0125` resumes strictly from
the latest Storage checkpoint, step 100; its step-150 failed tree remains
evidence and is not a resume source.

An HF-only export remains at `global_step_<n>/actor/huggingface` for M5, but
resume discovery now requires the matching `data.pt`, model state, and optimizer
state. This prevents a higher held-out candidate from being treated as a
resumable native checkpoint. Focused retention/capacity tests and the full
suite passed before the H100 resume is permitted.

The resumed run reached and published step 150 without error. Its next public
records began at step 151, proving that the append-only metric bridge
de-duplicates the replayed 101--150 range while continuing to publish new
training evidence. The current retention layout is older HF exports plus the
latest step-150 native resume payload.

The same resumed M3 then reached step 400 and published its final artifact.
Retention now consists of eight scheduled HF exports (steps 50--400) plus the
single step-400 native resume payload. That candidate set is complete for M5;
the next trainer action is the separate 200-step 0.5B deterministic
random-reward control in `d4-m4-0p5b-random-v0126`, never a held-out claim.

That control completed all 200 steps and published its final Storage checkpoint,
curve, and model manifest. Its only post-completion exception was the retained
Ray DataLoader-worker `Killed` teardown traceback after the runner's final
publish/`Finished` record; no CUDA/OOM, bridge, or entropy stop occurred. The
control is therefore evidence of the specified random-reward path only, not a
model-quality measurement or a substitute for M5.
