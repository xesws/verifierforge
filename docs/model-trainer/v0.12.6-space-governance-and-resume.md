# v0.12.6 — Checkpoint retention and resumable M3

The bridge will publish each completed native checkpoint through Storage, then
retain only the newest complete native payload for resume. For every older
checkpoint it retains `global_step_<n>/actor/huggingface` and removes native
optimizer/model/data payloads, so M5 can evaluate every scheduled export.

Before a child launches, the trainer asks the bridge/storage budget helper for
the effective checkpoint count and peak bytes. A known resume checkpoint gives
measured native and HF sizes; a fresh run falls back to cached model bytes with
a conservative native estimate. The helper rejects projected demand above 80%
of free filesystem space and prints the calculation.

The v0.12.5 M3 wrapper failed while its child ran on. v0.12.6 treats bridge
publication failure as terminal: it records the error, preserves the failed
staging tree under evidence, terminates the child process group, and returns a
nonzero result. After S2 cleanup, `d4-m3-1p5b-r1-v0125` resumes strictly from
the latest Storage checkpoint, step 100; its step-150 failed tree remains
evidence and is not a resume source.
