# v0.4.0 — Model Trainer: GRPO Smoke

## Scope

Run Qwen2.5-0.5B-Instruct through verl 0.8.0 GRPO on a 40/10 reviewed V1
NL-to-SQL split. Use one L4-safe configuration: LoRA rank 16, four prompts,
eight vLLM rollouts, 1,024 prompt tokens, 512 response tokens, validation every
10 steps, and checkpoints every 50 steps. A single OOM retry may reduce to two
prompts, 256 response tokens, and 0.35 vLLM memory utilisation.

`vf train <preflight-job> grpo_v1_0p5b_preflight` is the two-step GPU-only
preflight entrypoint; it checkpoints at step 2 and must use a separate job ID.
The real 100-step run uses `grpo_v1_0p5b`, retaining the step-50 checkpoint
cadence and the subsequent kill/resume proof.

## Persistence and Resume

verl writes a staging `global_step_<n>` checkpoint under `runs/<job>/.verl-staging`.
A sidecar publishes each completed step through `LocalStorage.save_checkpoint`;
the launcher resumes only from `Storage.load_latest_checkpoint`, then passes the
nested `global_step_<n>` directory to verl. Metric bridging maps file-logger
reward, validation accuracy, and entropy into append-only `MetricRecord` lines.

## Non-goals and Validation

This is a 100-step path smoke, not a quality claim. Stop an unmoving curve after
100 steps. Validate a two-step preflight, checkpoint at 50, kill/resume at 50,
final checkpoint at 100, and synced local `GET /jobs/<id>/metrics`.

## Acceptance Evidence — 2026-07-15

`d2preflight` completed its two-step GPU-only path and published step 2. The
real `d2demo` run published `step_50`, was stopped with `vf kill`, resumed from
`runs/d2demo/ckpt/step_50/global_step_50`, and finished after publishing
`step_100`. The first new public metric after resume was step 53, and the final
public JSONL had 100 unique ordered rows. This evidence is an operational smoke
only; the observed final validation accuracy of 0.60 is not a gain claim.
