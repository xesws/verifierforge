# v0.11.2 — Trainer: stack-level Blackwell smoke diagnosis

The diagnostic run uses the unchanged `grpo_v1_1p5b_blackwell_smoke`: 1.5B,
one GPU, TP 1, `k=8`, `enforce_eager=true`, and
`VLLM_ATTENTION_BACKEND=TORCH_SDPA`. Its only purpose is to reproduce a
three-minute `WorkerDict` stall and reveal the exact Python execution frames.

After reproduction, `py-spy dump` is taken from the launch runner, the verl
driver, and every Python process in their Ray/verl session. The run sheet holds
the unabridged output so a conclusion can distinguish a compiled extension,
an import path, a lock wait, or a non-Python/native wait.

A trainer mutation is prohibited unless a raw stack makes one narrow remedy
unambiguous. A JIT compilation frame may permit disabling/removing that
extension; an import frame may permit one documented environment bypass. Before
such a change, update the v0.11.2 documents and run sheet; then do one retest
only. A second stall or the 20-minute total deadline ends the diagnostic and
does not admit M3–M6.

**Authorized repair:** the diagnostic's raw vLLM error says
`VLLM_ATTENTION_BACKEND=TORCH_SDPA` is unsupported by V1 and forces a V0
fallback, while verl invokes V1 `AsyncLLM`; it then fails with
`envs.VLLM_USE_V1=False`. The clear, vLLM-recommended one-step action is to
remove only this `TORCH_SDPA` environment injection by setting the smoke config
field to `null`. The repair does not choose a `VLLM_USE_V1` value, alter the
model/data/reward/TP/memory settings, or authorize another fallback.
