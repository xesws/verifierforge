# v0.32.1 — Supabase API provenance check

The self-check will label every frozen operation as `SUPABASE`, `MIXED`,
`ARTIFACT`, or `N/A`; an HTTP 200 alone is not proof because cluster catalog and
job routes have deliberate local/static fallbacks. Supabase proof requires
direct repository readback of the same identifier or value and confirmation
that the SQLite sentinel path was not created.

Write operations use scoped fixtures: a generated queued Job, a synthetic
Nebius credential owner, idempotent routing/sample-source values, and a mock
Agent decision/approval chain. Before the run, current route and sample-source
records are captured. Afterward, test rows and any mock trace are removed and
the captured product state is restored. Secrets, DSNs, ciphertext, and API keys
must not enter output or tracked evidence.

Result: direct Postgres readback matched the generated Job, route,
sample-source, Agent decision/approval, Guardian curve, and synthetic Nebius
credential. Cleanup left zero self-check Jobs, credentials, or approvals and
no SQLite sentinel. Cluster catalog reads remain a documented mixed/static
path and update catalog timestamps; D4 hybrid Job detail remains artifact
sourced rather than being mislabeled as Supabase.

The shared-Postgres contract test also stopped assuming deleted sequences
restart at one and now isolates expected failures with SAVEPOINTs. This keeps
the suite rollback-only against populated Supabase while preserving the same
production repository implementation.
