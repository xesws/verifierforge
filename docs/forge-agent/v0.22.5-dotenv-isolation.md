# v0.22.5 — Non-mutating dotenv resolution

**Status:** complete — 2026-07-18
**Owner/worktree:** Codex, repository `main`

## Problem and scope

`LLMSettings.from_env()` and Gate C's live settings loader call
`load_dotenv()`, which mutates `os.environ`. During the complete test suite,
an LLM settings test can therefore load the developer's real
`VF_DB_BACKEND=postgres`; a later routing test then writes its 50% canary
fixture into Supabase instead of isolated SQLite. This patch will read dotenv
values into a local mapping, merge shell values over those defaults, and leave
the process environment unchanged.

## Non-goals

Do not change provider selection, model policy, public contracts, routing
semantics, or the developer's ignored `.env` file.

## Affected code and contract impact

- `app/gpt/client.py`: non-mutating dotenv resolution for shared LLM settings.
- `app/agent/evaluator.py`: the same behavior for Gate C live settings.
- Focused tests prove dotenv defaults still work, shell values still win, and
  unrelated variables never appear in `os.environ`.

There is no contract or database schema change.

## Validation plan

Run the focused LLM, evaluator, and routing tests. Set the real canary to zero,
run the complete suite, then query Supabase directly and require that the
canary remains zero. Finish with shell syntax, secret scanning, and diff checks.

## Result

Both settings loaders now use `dotenv_values()` and merge the current shell
over local defaults without writing either mapping into `os.environ`. Focused
coverage passed `48 passed`; the complete suite passed `366 passed, 1 skipped`.
The real route was set to zero before the suite and remained zero afterward by
both direct Supabase query and API readback.
