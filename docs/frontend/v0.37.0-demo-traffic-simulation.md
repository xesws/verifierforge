# v0.37.0 Ship traffic simulation control

Ship adds `Simulate traffic (200 requests)` beside the Live Guardian surface.
The page reads task status on entry, adopts a run started by another browser,
polls it once per second while active, displays `sent/total`, disables duplicate
clicks, and restores the button after completion.

Traffic state belongs to `ShipPage`, not the empty/chart child components, so
the first Guardian point can replace the empty state without losing progress.
Guardian retains its normal 30-second polling, polls every five seconds while
traffic runs, and reloads immediately at completion. A cold serving state adds
the exact explanatory line:

`endpoint cold — traffic will route to default; wake the model to see tuned scoring`

The client uses the existing session-only Basic invitation. A feature-disabled
404 hides the optional control and leaves routing, reports, Wake, and the tuned
reviewer probe unchanged.

The page-flow test covers the exact cold copy, disabled live progress, task
completion, and empty-to-one-point Guardian transition. All 32 frontend tests,
ESLint, and the production build pass.

Vercel production deployment `dpl_5PAUihiKB6uncDadX3dpRPKFdB94` is Ready and
aliased to `https://verifierforge-web.vercel.app`. The public Ship route is 200,
its served JavaScript contains the button plus exact cold copy, and Railway
allows the stable origin for authenticated GET and POST preflight. During the
ready batch the public task/Guardian endpoints exposed the first new point at
46/200, which is the same transition exercised by the no-refresh page-flow
test. The session had no discoverable in-app browser, so no screenshot or
manual visual-inspection pass is claimed.
