# v0.18.0 — static product demo frontend

**Status:** implemented and browser-validated
**Owner / worktree:** Codex / repository-root `main`
**Version reservation:** `docs/versions/v0.18.x/v0.18.0-static-frontend.md`

## Problem and scope

Build a standalone, recording-ready VerifierForge product prototype in
`frontend/`. The implementation uses Vite, React, strict TypeScript, React
Router, Recharts, Lucide React, local font packages, and restrained CSS motion.
It contains Discover, Forge, Run, Prove, and Ship/Guard surfaces, with all
interactions running entirely in the browser.

## Explicit non-goals

- No runtime backend or network integration, SSR, authentication, database,
  remote endpoint, or backend-file modification.
- No fabricated, smoothed, or extrapolated training, control, held-out, or
  guardian evidence.
- No presentation of projected savings or arena samples as verified results.

## Affected code and documentation areas

- `frontend/src/app/`: routing and application shell.
- `frontend/src/pages/`: the five product surfaces.
- `frontend/src/components/`: glass, navigation, metric, evidence, chart,
  comparison, routing, and guardian primitives.
- `frontend/src/data/`: generated artifact evidence and separate typed product
  scenario fixtures.
- `frontend/src/styles/`: tokens, layout, component, motion, and responsive CSS.
- `frontend/scripts/`: deterministic artifact-to-fixture generation.
- This document and the v0.18.0 version record.

## Contract impact

None. The frontend is statically isolated and has no backend client. Existing
artifact schemas are read by a build-time script without changing the sources.

## Static data strategy and evidence boundary

The generator reads both committed metrics JSONL files line by line and
preserves every `step`, `reward_mean`, `pass_at_1`, `entropy`, and `timestamp`
value. The committed generated JSON keeps the application buildable after the
frontend directory is copied away from the repository. Held-out checkpoints,
rows, pass@1, pass@8, selected step, model, and verdict are verified evidence.

`productScenario.ts` contains the three commercial clusters, projected monthly
savings, projected arena win rate, and seven demo SQL comparisons. The UI labels
the last three concepts as projections or demo samples. Training-monitor
pass@1 and held-out pass@1 remain separate charts and explanatory concepts.

## Five-page user flow

- `/discover`: compare all clusters; inspect the live SQL gain or forge either
  discovered cluster.
- `/forge/new`: edit a URL-prefilled local form, queue a job in localStorage,
  and open its locally rendered detail.
- `/jobs/d4-m3-1p5b-r1-v0125`: inspect quality, reward, and entropy using all
  raw metrics, with the control ending at step 200 and step 350 marked.
- `/reports/d4-m3-1p5b-r1-v0125`: review held-out gain, checkpoint selection,
  projected savings, and clearly labeled demo arena samples.
- `/ship/data-pull-sql`: change local routing, canary, and target state while
  guardian evidence waits for sampled canary traffic.

## Responsive requirements

At 1440×900 the Discover clusters share one viewport and charts use their full
horizontal range. At 1024×768 the navigation and panels compact with preserved
hierarchy. At 390×844 the navigation moves to a bottom rail, panels stack,
forms remain touch-friendly, charts retain readable labels, and the document
does not overflow horizontally.

## Visual acceptance criteria

The workspace uses bright floating glass over fog, ice, and mint light fields;
soft white borders; low-contrast shadows; 24–30 px primary panel radii; Manrope
UI copy; and IBM Plex Mono for evidence labels. Cost leads Discover hierarchy,
the report hero leads proof hierarchy, and chart panels receive more space than
supporting metrics. There is no logo image, purple/black cyber theme, neon,
heavy shadow, or default component-framework aesthetic. Motion runs once and
respects reduced-motion preferences.

## Validation plan

Install dependencies; regenerate data; lint strict TypeScript and React code;
build production assets; then verify all required desktop, tablet, and mobile
viewports in a browser. Check interactions, persistence after reload, chart
accessibility, keyboard focus, console output, evidence labels, exact public
values, and screenshot composition. Confirm no runtime network primitive exists
in frontend source and no backend file changed.

## Validation result

The generated evidence fixture contains exactly 400 main points and 200 control
points; the main finishes at step 400 and the control naturally finishes at
step 200. The selected step and held-out before/after values match the committed
report. A source scan found no `fetch`, Axios, WebSocket, or XMLHttpRequest use.

Data generation, ESLint, strict TypeScript compilation, and the Vite production
build pass. Browser checks at 1440×900, 1024×768, and 390×844 found no horizontal
overflow or console error. The Forge queue and routing switch persist after
reload, the Arena control expands all seven demo samples, and the mobile chart
retains readable axes, legend, control termination, and checkpoint marker.
Screenshots are stored in `frontend/screenshots/` for desktop Discover, desktop
training, desktop proof, and mobile Discover.
