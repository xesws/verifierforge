# v0.35.0 — Frontend API integration

Nora's Vite/React frontend remains the presentation system. The integration
adds a small typed data layer and missing product controls without changing
her typography, glass panels, color tokens, navigation, chart library, or
responsive layout conventions.

## Runtime data boundary

`VITE_VF_API_BASE_URL` is the only API origin. The invitation code is entered
at runtime, sent as HTTP Basic user `judge`, and retained only in browser
session storage. The frontend never contains an invite code or provider key.

The 22 frozen operations are exposed through five dependency-gated pages:

- Discover owns clusters, sample Input, Analyze, and the advisory decision.
- Forge owns the proposed config, approval, explicit Start boundary, completed
  reference-run handoff, queued Job controls, and write-only provider settings.
- Runs owns job status plus the 400-step main and 200-step control curves.
- Proof owns held-out before/after, the two-curve proof, ten arena samples,
  savings formula/assumptions, and verdict.
- Ship owns routing, canary, Guardian points, serving Wake/status, and the
  ready-only tuned probe.

Progress is stored in versioned `sessionStorage`, not inferred from shared
historical records. Analyze must return `forge` with a config before Forge
unlocks; an approval plus an explicit actual/reference job selection unlocks
Runs; a completed report unlocks Proof; an explicit proof acknowledgement
unlocks Ship. Direct links fail closed to the earliest incomplete stage, and
leaving the invitation session clears the journey.

Start Forge is not an alias for creating Job metadata. Approve records intent
without spend; Start Forge is a separate literal confirmation. Its expected
public-reviewer result is an explicit disabled state while
`VF_AUTOPROVISION=false`.

## Failure behavior

Each resource renders loading, empty, error, readonly, or cold state inside
the existing page rather than crashing the application. Serving cold or a
tuned fallback never hides the immutable Job/report evidence. Mutating calls
are never retried automatically.

The browser transport binds the native fetch receiver to `window`. Tests must
fail if a future refactor invokes native fetch with the API client as `this`;
this is required for WebKit compatibility.

Wake progress is a client-observed registry activity log. It uses only real
`state`, `detail`, and `updated_at` changes from `/serving/status`, plus elapsed
time and the measured 267–282 second cold-start range. It must not claim to be
raw pod output or invent a continuous download percentage.

`POST /serving/tuned-completion` bypasses production canary selection and
directly resolves the ready tuned endpoint. It never changes routing state and
never falls back. The UI shows the request lifecycle and the complete returned
SQL with model, request ID, finish reason, usage, latency, and verified route.

## Acceptance

Client tests enumerate all 22 method/path pairs. Component tests cover auth,
cluster decisions, approval/start separation, chart lengths, report evidence,
routing, Guardian, route guards, serving lifecycle, and tuned-only completion.
Browser evidence is
captured at desktop and mobile widths against the Railway-backed build.

The companion `scripts/frontend_contract_smoke.py` exercises the frozen
operation set without printing its Basic invitation or provider credentials.
The paid serving Wake remains separately armed and is never part of an
ordinary contract smoke.
