# v0.35.4 — Ship live SQL execution

Ship becomes a deliberate two-step playground:

1. **Generate SQL with tuned model** calls the ready tuned endpoint and shows
   the exact OpenAI-compatible model response.
2. **Run SQL on frozen demo data** executes that exact response in a local
   SQLite/WASM sandbox and shows the real table or error.

The page must display this sentence beside the execution action:

> queries run against the frozen demo dataset (the same schema the verifier trained on)

The primary runtime preloads a same-origin `sql.js` WASM worker and the frozen
synthetic schema asset when Ship mounts. Each click creates and closes a new
in-memory database. No execution request goes to Railway, Supabase, the GPU, or
an LLM. Once SQL has been generated, it remains executable after serving goes
cold.

Only one complete plain statement or one complete Markdown SQL fence is
accepted. The runtime requires one SELECT/WITH statement, turns on SQLite
query-only mode, rejects dangerous keywords, caps input at 10,000 characters,
returns at most 100 rows, and terminates a query after two seconds. Worker
messages drive the visible initialization, fixture verification, database
creation, execution, and terminal states; the UI does not interpolate fake
progress.

The generated-SQL card and execution-result card remain separate. Success
shows real columns, values, NULLs, row count, truncation, duration, execution
identity, timestamp, SQLite version, and dataset hashes. Rejection, SQLite
errors, timeout, and runtime failure each have explicit messages and preserve
the generated SQL for Copy or Retry. A new prompt or completion clears the old
execution result.

This change extends the existing Ship experience and is not a new autonomous
subsystem, so it does not add a feature flag. The reviewer still must complete
the existing Discover → Forge → Runs → Proof → Ship dependency gates.
